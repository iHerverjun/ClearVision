# 系统配置功能审计报告（最终核查版）

- 审计日期: 2026-02-27
- 审计范围: `Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/settings/settingsView.js` 对应的“系统配置”页面及其后端调用链
- 审计方式: 静态代码深度追踪（前端控件 -> 事件绑定 -> API -> 持久化 -> 运行时消费）
- 审计目标: 区分“真实有效功能”与“占位伪代码/后续待开发功能”，发掘隐患并提供优化建议
- 约束: 本次仅审计，不改代码

---

## 1. 判定标准

### 1.1 真实有效

满足以下条件:

1. 前端存在可触发交互（按钮/输入事件）。
2. 事件绑定存在，且能调用后端 API。
3. 后端存在对应接口与逻辑。
4. 数据被持久化，且在运行时被实际消费（不仅仅是保存）。

### 1.2 半有效

满足以下之一:

1. 交互和保存链路存在，但运行时未消费。
2. 前端局部生效（例如只写 localStorage），未纳入统一配置闭环。
3. 部分字段可保存，其他字段仍是硬编码。

### 1.3 占位伪代码

满足以下之一:

1. 仅有 UI 元素，无事件绑定。
2. 仅静态示例数据，无真实数据源/硬编码。
3. 无后端 API 或无运行时落地逻辑。

## 2. 执行摘要

1. 当前“PLC 通讯配置”以占位 UI 为主。除协议/IP/端口可随“保存所有更改”写入配置外，连接测试、地址映射、同步配置等均未形成真实闭环。

   - `AI 大模型管理`
   - `相机管理`

3. 大量设置项处于“可保存但不生效”或“前端硬编码覆盖”的半完成状态，尤其在:
   - 常规设置
   - 文件存储
   - 执行与运行
4. **新增隐患发现**: 全局保存时存在将关键字段（如活动相机 ID）置空的逻辑硬伤，并在心跳间隔、磁盘阈值等存在强制重写硬编码的问题。

## 4. 详细审计结果

### 4.1 全局保存

#### 4.1.1 保存所有更改

- 判定: 半有效 (高风险)
- 前端证据:
  - 按钮渲染: `settingsView.js:140`
  - 监听绑定: `settingsView.js:206-208`
  - 保存实现: `settingsView.js:1755-1840`
- 后端证据:
  - 获取/保存设置接口: `SettingsEndpoints.cs:26-43`
  - 相机绑定保存接口: `SettingsEndpoints.cs:204-233`
- 风险项:
  - **并发写一致性**: 多步骤保存存在部分成功窗口参数断层的可能（非事务式一致性）。
  - **活跃状态清空陷阱**: 收集对象的 `activeCameraId` 被硬编码赋予了 `''` (settingsView.js:1812)，保存触发后会清掉全局唯一当前相机凭证。
  - **脏模型风险**: L1721 `collectCameraBindings()` 仅返回了原内存数组引用，并没有重新从 DOM 读取或深拷贝，导致依赖关系脆弱。
  - **隐式 AI 表单保存**: L1826 调用了 `_saveCurrentForm()` 保存正在编辑状态下的 AI 模型表单项。

#### 4.3.1 协议/IP/端口输入

## 6. 安全与系统隐患级联评级

### 6.1 🔴 高风险（可能引起直接系统工作反常）

1. **`activeCameraId` 被保存强制清空问题**: 多保存几次主页面，会导致后续模块寻址无法找到唯一的当前活动相机设备。
2. **`heartbeatIntervalMs` 硬重写风险**: 使用默认配置面板保存会导致其他手段维护的 PLC 通讯配置数据突发变质受损。
3. **前端强覆盖问题**: 文件存储模块中关于清理阈值 `retentionDays/minFreeSpaceGb` 没有通过DOM ID读取，反而锁死了数值 30与 5 推发至后端。

### 6.2 🟡 中风险

1. PLC 配置页面“看上去能调、实际不运转”的设计迷惑性。
2. 全局保存使用分别向多终点依次调用的策略。
3. AI 设置表单随着 `_saveCurrentForm` 被隐性包裹自动在全局按钮触发中提交，这可能引起当前测试/草稿信息不当落地。
4. 重复同名函数 (`bindUserManagementEvents`) 和地址映射内的静态示意。

## 7. 建设性改进建议（包含执行优先度排序）

### 第一梯队：修正硬编错误与反常覆写逻辑

*   **修复活动相机覆写**: 在 `save()` 方法组装 JSON 前读取维护好的 `this.cameraBindings` 结构，补充赋予原活跃主 ID，不能直接设定为 `''`。
    *   **状态**: ✅ 已完成（2026-02-27）
    *   **落地证据**: 新增并统一使用 `resolveActiveCameraId()`，`saveCameraBindings()` 与 `save()` 均改为基于现有绑定解析活动相机 ID，不再写死空字符串（`settingsView.js:1720, 1729, 1777, 1826`）。
*   **纠正强制锁死的默认覆盖属性**:
    *   移除 `retentionDays` 与 `minFreeSpaceGb` 这俩项发出的前置强行锁定；给 HTML Input 添加相应 `id` 并从该 `id` 内读出 `value` 后解析。
        *   **状态**: ✅ 已完成（2026-02-27）
        *   **落地证据**: 存储页阈值输入已补充 `id`，保存时改为读取并解析用户输入，未填写时才回退配置/默认值（`settingsView.js:1139, 1147, 1769, 1773, 1818-1819`）。
    *   对于 `heartbeatIntervalMs`，可将其修改为从 `this.config.communication.heartbeatIntervalMs`（加载时的原本值）提取填入作为保存透传。
        *   **状态**: ✅ 已完成（2026-02-27）
        *   **落地证据**: 保存时改为解析并透传 `this.config.communication.heartbeatIntervalMs`，保留正整数校验与默认值兜底，不再强制写死 `1000`（`settingsView.js:1765-1768, 1813`）。
*   **清理同名死代码**: 直接铲除 `settingsView.js` 在行 877 左右的 `bindUserManagementEvents` Dummy占位法实现。
    *   **状态**: ✅ 已完成（2026-02-27）
    *   **落地证据**: 占位实现已删除，当前仅保留一处真实 `bindUserManagementEvents()` 实现（`settingsView.js:1573`；调用点 `settingsView.js:216`）。

### 第二梯队：填平业务空白层，推动局部可用化

*   **隔离隐式保存或加入事务流管控**: 合并设定存储过程或者向 UI 弹窗警告"部分模块将同步更新"（包含目前正开启编辑但不完整的 AI 模型输入面板）。
*   **让 `SavePolicy` 与 `ImageSavePath` 真正发挥效用**: 在核心 `ImageProvider` 或持久化流水线中接入判定该两项 `AppConfig` 的引用源，取代硬编码路径。
*   **实现真正的保存全局隔离与连接测试**: 加入真正的 ping 测试或是后端探活请求支持真正的连点校验回访。

### 第三梯队：填补占位与长远基建发展

*   **连接配置统一源**: 当平台算子运作期间，引导其默认向 `AppConfig.Communication` 读取 PLC 的通讯握手预备凭证。
*   **消除所有占位元素影响**: 对于还未排期推进的功能项目（类似"清理功能"、"修改容量警告界限"、"动态安全密码区"和"动态图表"），通过临时隐藏该 UI 部件或者弹窗提醒“开发中”的方式抹除误导风险。

## 8. 审计结论（最终）

当前“系统配置”页属于**“混合成熟度较高，但多源头分离”的融合型界面**:

1. `AI 大模型池`、`多平台相机硬件绑定流` 及 `系统用户安防分配` 等模块已处于**高成熟度，且具备安全意识与防脱保设计**。拥有良好的 CRUD 打通环境。
2. `PLC 通讯配置与控制映射` 基本可以评估为**UI壳层产物**（实质控制多散布于其他流程管道）。
3. 其余系统运维模块（常规/存储/控制阈值）呈现出最棘手的**能更改、存盘但是并未驱动内部系统工作的特性**。甚至包含高风险反向强写入属性逻辑。

建议开发团队**首先针对 第一梯队（如 `activeCameraId` 被清空及硬编码）的问题进行速修**，随后通过评估，逐步向配置主推驱动底层实际运算过程发力建设。
