# 线序检测结果可观测性与 NMS 收口执行记录

日期：2026-03-23

## 背景

当前端子线序模板已经具备固定骨架，但现场调试仍有两个主要痛点：

- 结果侧只看到 `Message`，无法快速判断是“没检出、重复框、错序还是阈值过严”。
- `DeepLearning` 内部已有一轮 NMS，下游 `BoxNms` 又做一轮，`DetectionSequenceJudge.MinConfidence` 还会继续过滤，阈值职责分散。

## 本轮思路

本轮不先做大 UI，而是先把链路结果说清楚，再把阈值和 NMS 权限收口到清晰的一层。

### 1. 结果可观测性增强

- `BoxNms` 输出结构化诊断：
  - 输入数量
  - 分数阈值过滤后的候选数量
  - 保留数量
  - 抑制数量
  - 抑制框列表
- `DetectionSequenceJudge` 输出结构化诊断：
  - 是否匹配
  - 预期顺序 / 实际顺序
  - 原始数量 / 过滤后数量 / 排序后数量
  - 缺失标签 / 重复标签
  - 说明消息
- `ResultOutput` 同时接收图像、NMS 诊断、线序诊断和文本消息，形成一次运行的统一诊断输出。

### 2. NMS / 阈值收口第一阶段

- 在线序模板中关闭 `DeepLearning` 内部 NMS，让 `BoxNms` 成为唯一的框去重 owner。
- 在线序模板中把 `DeepLearning.Confidence` 降为低置信度地板，不再作为首要现场调参项。
- 在线序模板中把 `DetectionSequenceJudge.MinConfidence` 固定为 `0.0`，避免第三层过滤。
- 现场优先调参收口到：
  - `BoxNms.ScoreThreshold`
  - `BoxNms.IouThreshold`

## 执行计划

## 补充修复：图像采集节点预览复用旧图

### 现象

- 在流程里执行过一次推理后，再预览 `ImageAcquisition` 节点并点击“刷新”，预览不会重新采图，而是重复显示上一轮推理结果图。

### 根因

- 节点预览前端会把最近一次检测结果图作为 `inputImageBase64` 传给预览接口。
- `ImageAcquisitionOperator` 的设计中，只要收到外部 `Image` 输入，就优先走“透传模式”，不会重新从相机或文件采图。
- 因此，`ImageAcquisition` 上游本应重新采图的预览请求，被外部注入的旧图短路了。

### 修复策略

- 在 `PreviewNodeEndpoints` 中增加判断：
  - 如果目标节点的上游子图包含 `ImageAcquisition`，则不注入外部 `inputImageBase64`。
  - 只有在整个上游路径中不存在采图节点时，才允许使用外部图像作为预览输入。

### 回归验证

- `PreviewNodeEndpointsTests` 已迁移到新 DTO：
  - `UpdateFlowRequest`
  - `OperatorDto`
  - `PortDto`
  - `OperatorConnectionDto`
- 已新增正式回归：
  - 有 `ImageAcquisition` 上游时，不得注入外部图像
  - 没有 `ImageAcquisition` 上游时，仍应保留外部图像注入能力

## 补充修复：文件采集模式下 ROI 预览误判为相机模式

### 现象

- 上游 `ImageAcquisition` 已填写 `FilePath`，但 `RoiManager` 预览仍提示“请先选择相机”或“打开相机预览”。

### 根因

- 预览链路和 `ImageAcquisitionOperator` 都把 `SourceType=Camera` 视为最高优先级。
- 一旦用户已经选择了文件，但 `SourceType` 还停留在 `Camera`，预览执行会优先走相机分支并失败。

### 修复

- `ImageAcquisitionOperator`：只要存在显式 `FilePath`，文件模式优先于 `SourceType`。
- `previewCoordinator`：前置校验中，`FilePath` 存在时直接视为满足预览前提，不再要求相机。
- `PropertyPanel`：当 `ImageAcquisition` 通过文件选择器选中文件时，自动把 `SourceType` 切换到 `File`。

### 验证

- `dotnet test Acme.Product/tests/Acme.Product.Tests/Acme.Product.Tests.csproj --filter "FullyQualifiedName~ImageAcquisitionOperatorTests" --nologo`
- 结果：`5 passed, 0 failed`

## 补充修复：线根检测预览中的重叠框与问号

### 现象

- 在线序模板中，`DeepLearning` 节点预览会看到大量重叠框。
- 预览图顶部统计文字显示为问号。

### 根因

- 模板已显式配置 `EnableInternalNms=false`，因此 `DeepLearning` 节点输出的是原始候选框，业务语义上由下游 `BoxNms` 才做唯一 NMS。
- `DeepLearningOperator.DrawResults` 使用 OpenCV `PutText` 直接绘制中文统计文本，Hershey 字体不支持中文，因此显示为问号。

### 修复

- 保持 `DetectionList / Objects` 仍输出原始候选框，不改变线序模板“`BoxNms` 是唯一业务去重 owner”的语义。
- 当 `EnableInternalNms=false` 时，仅对 `DeepLearning` 的预览图做一轮“视觉用途”的去重后再绘制，避免预览中一屏重叠框。
- 将预览图统计文字改为 ASCII：
  - `Objects: N`
  - `Defects: N`
- 前端诊断卡片中新增 `预览去重说明`：
  - `原始候选框`
  - `预览图显示框`
  - `预览去重层`
  - `业务去重层`
  - `内部 NMS`
  - 明确说明“Preview-NMS 仅用于可视化，真正业务去重仍由 BoxNms 执行”

- `dotnet test Acme.Product/tests/Acme.Product.Tests/Acme.Product.Tests.csproj --filter "FullyQualifiedName~DeepLearningOperatorTests" --nologo`
