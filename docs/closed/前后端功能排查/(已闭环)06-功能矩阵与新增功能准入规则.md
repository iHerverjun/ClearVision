## 当前架构结论

1. 主 UI 中已经下线的入口，不允许以“临时联调方便”为理由重新挂回公开界面。
2. 已经确定的主协议，不允许和新协议长期并行扩散；兼容层只能收敛、不能继续长大。
3. 结果页、相机状态和能力状态 registry 属于当前三块重点维护区域，任何新增需求都不能绕开这三个结论。

## 新增功能准入规则

### 1. 主入口规则

1. 每项业务能力只能有一个公开主入口。
2. 如果因为迁移期保留兼容入口，必须在功能矩阵里明确“谁是主入口、谁是退场入口、预计何时删除”。
3. 已从公开界面下线的入口，后续只能做归档或删除，不能偷偷恢复。

### 2. 主协议规则

1. 每项业务能力只能有一条主协议路径。
2. 兼容协议只允许作为短期回退或迁移层存在，必须有删除方向，不能没有结束时间。
3. 新增接口前先检查现有接口是否已经能表达同一语义；能复用就不新增。

### 3. 真实状态规则

1. 页面显示的状态必须能追溯到真实后端数据、真实设备状态或真实执行结果。
2. 禁止再新增“假连接状态 / 假成功提示 / 演示值回退 / 仅前端记账”的实现。
3. 如果真实能力尚未接入，界面必须明确标记“未接入”或禁用，而不是伪造正常状态。

### 4. 回归与验证规则

1. 触达主链路的改动，必须同时更新 [`05-联调回归基线.md`](./05-联调回归基线.md) 中的适用项。
2. 至少提供一种可复用验证资产：自动化测试、最小手工联调步骤或明确的未验证说明。
3. 构建环境不干净时，必须在文档里保留阻塞说明，不能把“未执行”写成“已通过”。

### 5. 文档同步规则

1. 改动如果改变了公开入口、主协议、主数据流或禁用策略，必须同步更新本目录对应阶段文档。
2. 功能状态只能写“已完成 / 部分完成 / 未完成”，并与当前证据保持一致。
3. TODO 文档用于跟踪阶段状态，功能矩阵用于给新改动设边界，二者需要一起维护。

## 新增功能评审最小清单

| 检查项 | 必答问题 |
| --- | --- |
| 入口 | 这次改动的唯一公开入口是什么？是否会新增第二入口？ |
| 协议 | 是否复用了现有主协议？如果保留兼容层，退场计划是什么？ |
| 状态 | 页面展示的每个关键状态能否追溯到真实来源？ |
| 禁用态 | 如果后端能力未接入，界面如何明确禁用或提示？ |
| 回归 | 需要更新哪些基线项、测试或手工联调步骤？ |
| 文档 | 是否同步更新了本目录的阶段状态、功能矩阵或准入规则？ |
