# 前端架构与交互缺陷审计

- 基线日期：2026-03-20
- 审计方式：主代理整合多名 `gpt-5.4` 子代理审计结果，结合本地静态代码核查整理
- 审计范围：
  - `Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js`
  - `Acme.Product/src/Acme.Product.Desktop/wwwroot/src/core`
  - `Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features`
  - `Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components`
- 目标：
  - 识别是否仍存在与 `handEyeCalibWizard.js` 类似的“自建弹层 + 全局 DOM + 生命周期不对称”问题
  - 固化前端当前的架构单点、生命周期风险和交互设计缺陷
  - 为后续 Phase 3 架构收敛提供优先级明确的整改入口

## 结论概览

### 最高优先级结论

1. 当前前端最大的单点仍是 [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js) 的顶层模块导入链。
2. 当前前端最容易重复出现的同类问题仍是“组件绕过共享基元，自建弹层/全局事件/全局实例，然后缺少对称销毁”。
3. 当前最容易让用户卡住的路径集中在“认证跳转”“设置页相机管理 -> 预览/标定”“结果页筛选/分页语义”三条主链路。

### 风险分组

| 严重度 | 代表问题 | 类型 |
| --- | --- | --- |
| `P0` | 单个前端模块编译失败导致整站假死 | 架构单点 |
| `P0` | 认证启动链是同步守卫 + 异步补刀，壳层可能先露出 | 启动链 / 状态时序 |
| `P1` | 旧标定向导、设置页用户弹窗、自建向导绕过共享 modal | 生命周期 / 弹层 |
| `P1` | `window.*` 全局实例、`window.dispatchEvent` 事件总线、重复启动副作用 | 生命周期 / 全局副作用 |
| `P1` | 设置页、标定、预览、结果页的入口和语义不透明 | 交互设计 |

## 一、架构单点与启动阻断

### 1. `P0` 顶层模块导入链仍是前端单点故障

- 代表文件：
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L51)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L1790)
  - [`inspectionPanel.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/inspection/inspectionPanel.js#L7)
- 现象：
  - `app.js` 顶层直接导入核心与多个 feature 模块。
  - 任一 feature 模块在 import 阶段语法损坏，就会直接中断整站初始化。
  - 本轮已出现真实样本：`inspectionPanel.js` 中 stray `*/` 导致整站只剩静态壳。
- 后果：
  - 用户看到的是“界面卡死”“主壳不动”“无法跳登录”，但真实原因是某个 feature 模块导入失败。
- 短期止血：
  - 把认证壳层、设置页、检测页、结果页按视图懒加载。
  - 至少保证“登录跳转 / 基础壳层”不依赖高风险业务模块。
- 中期方向：
  - 把 [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js) 降级为壳层编排器。
  - 建立按 view route 拆分的入口，避免“一个文件 import 全站”。

### 2. `P0` 认证启动链仍是混合时序

- 代表文件：
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L106)
  - [`auth.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/auth/auth.js#L79)
  - [`auth.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/auth/auth.js#L100)
  - [`login.html`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/login.html#L117)
- 现象：
  - `initAuth()` 在模块加载时同步决定是否重定向。
  - `validateTokenAsync()` 又在 `initializeApp()` 内异步验证并可能再次跳转。
  - 登录页还会做自动回跳。
- 后果：
  - 主壳可能先出现，再异步跳登录。
  - 任何 feature 模块失败时，用户会误判成“卡死”而不是“认证失效”。
- 短期止血：
  - 把“本地会话存在性检查”和“服务端 token 校验”合并成单一 bootstrap gate。
- 中期方向：
  - 建立统一的 session bootstrap / auth state machine。

### 3. `P1` 视图切换承担了过多初始化职责

- 代表文件：
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L162)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L221)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L257)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L273)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L307)
- 现象：
  - `switchView()` 不只是切 DOM，还负责初始化检查页、图像查看器、结果页加载、设置页 refresh。
- 后果：
  - 容易重复绑定、旧实例残留、不同视图之间相互影响。
- 短期止血：
  - 把 `switchView()` 限定为“显示/隐藏 + 激活/停用”。
- 中期方向：
  - 每个 view 拥有单独的 mount/unmount 生命周期。

## 二、与手眼标定同类的生命周期问题

### 4. `Critical` 旧版标定向导仍是“全局 DOM + 全局单例”模式

- 代表文件：
  - 历史文件 `calibrationWizard.js`（已于后续收敛中删除）
  - [`settingsView.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/settings/settingsView.js#L831-L837)
  - [`handEyeCalibWizard.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/core/calibration/handEyeCalibWizard.js#L410-L470)
- 为什么同类：
  - 该问题最初来自旧版 `calibrationWizard.js` 的全局 DOM / 全局单例实现；当前仓库中旧文件已删除，公开入口已统一到实例化的 `HandEyeCalibWizard`。
- 后果：
  - 重复打开、旧 DOM 残留、ID 冲突、事件绑到旧实例。
- 建议：
  - 该项已进入“已退场”状态，后续只需继续避免恢复旧入口或重建全局单例模式。

### 5. `High` 设置页用户弹窗又手写了一套 modal 生命周期

- 代表文件：
  - [`settingsView.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/settings/settingsView.js#L2057)
  - [`settingsView.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/settings/settingsView.js#L2113)
- 为什么同类：
  - 这和手眼标定自建弹层同类，都是绕过共享 modal 基元，自己管理关闭和销毁。
- 后果：
  - 关闭行为和其他弹层不一致。
  - 一旦后续加对象 URL、计时器或外部监听，当前 `container.innerHTML = ''` 很容易漏清理。
- 建议：
  - 统一收敛到 [`uiComponents.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/uiComponents.js#L368) 的 `createModal`。

### 6. `High` 共享 modal 基元本身关闭契约不完整

- 代表文件：
  - [`uiComponents.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/uiComponents.js#L368)
  - [`uiComponents.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/uiComponents.js#L440)
- 为什么同类：
  - `handEyeCalibWizard.js` 的问题之一就是关闭路径不统一；共享基元自身没有完整关闭契约，业务组件就只能各自打补丁。
- 当前缺口：
  - 只有关闭按钮和点击遮罩关闭。
  - 没有统一 `Escape` 关闭。
  - 没有 `onBeforeClose/onDispose` 这类资源清理钩子。
- 建议：
  - 统一 `close button / overlay click / Esc` 三条关闭路径。
  - 补生命周期钩子，给对象 URL、SSE、消息监听、计时器释放提供正式出口。

### 7. `Medium` Tooltip / SplitPanel 仍是典型的全局监听生命周期风险

- 代表文件：
  - [`tooltip.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/tooltip.js#L34)
  - [`tooltip.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/tooltip.js#L92)
  - [`splitPanel.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/splitPanel.js#L67)
  - [`splitPanel.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/shared/components/splitPanel.js#L126)
- 为什么同类：
  - 虽然不是 modal，但本质仍是“组件生命周期脱离实例，依赖 `document/window` 存活”。
- 建议：
  - Tooltip 补 `destroy()`。
  - `splitPanel` 的 document/window 监听必须用实例字段保存 handler，确保 `destroy()` 能真正卸载。

## 三、全局副作用与重复绑定

### 8. `P0` 状态栏更新被启动了两次

- 代表文件：
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L135)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L1790)
- 现象：
  - `initializeApp()` 内启动一次 `startStatusBarUpdates()`。
  - `DOMContentLoaded` 回调里又启动一次。
- 后果：
  - `statusBarInterval` 虽会覆盖，但 `updateFPS()` 是自递归 `requestAnimationFrame`，会并行跑两条循环。
- 建议：
  - 只保留一个启动入口，并给 FPS 循环加幂等/停止保护。

### 9. `P1` `window` 级事件总线没有销毁路径

- 代表文件：
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L543)
  - [`projectView.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/project/projectView.js#L305)
  - [`projectView.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/project/projectView.js#L451)
- 现象：
  - `projectOpened` 用 `window.dispatchEvent(new CustomEvent(...))` 派发，`app.js` 监听后重置画布、切视图、清结果页。
- 后果：
  - 再初始化或热重载时会叠加监听器，核心业务副作用重复执行。
- 建议：
  - 收敛到 `projectManager.subscribe(...)`，不要再用 window 事件总线驱动核心业务。

### 10. `P1` `window.*` 全局实例耦合仍然很重

- 代表文件：
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L296)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L327)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L591)
  - [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js#L1275)
  - [`inspectionController.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/inspection/inspectionController.js#L279)
  - [`inspectionPanel.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/features/inspection/inspectionPanel.js#L719)
- 现象：
  - `window.flowCanvas`、`window.inspectionImageViewer`、`window.resultPanel`、`window.aiPanel`、`window.currentUser` 被跨模块直接读写。
- 后果：
  - 空引用、旧引用、跨视图串状态、初始化顺序耦合。
- 建议：
  - 收敛到单一 runtime registry 或 store。

## 五、最容易让用户困惑的 3 条主路径

1. 设置页相机管理 -> 搜索到设备 -> 添加绑定 -> 想做预览/标定  
问题在于“必须先选中一行”这个前提没有被界面明确告知。

2. 设置页相机管理 -> 点击“手眼标定向导”  
问题在于向导允许无上下文启动，用户会直接进入黑屏/占位提示状态。

3. 打开程序时会话已失效 -> 主壳先出现 -> 再跳登录  
问题在于用户感知是“卡死”而不是“认证失效”。

## 六、建议的整理顺序

1. 先处理 `P0`：
   - `app.js` 顶层 import 单点
   - 认证启动 gate
   - 启动副作用重复注册
2. 再处理同类生命周期问题：
   - 旧版标定向导退场
   - 共享 modal 契约统一
   - 设置页用户弹窗收敛
3. 最后处理高频用户路径：
   - 设置页相机管理 -> 预览/标定
   - 结果页筛选/分页语义
   - 检查页运行保护前置告知

## 七、总评

当前前端的核心脆弱点不是某一个按钮，而是三件事叠在一起：

1. [`app.js`](../../../Acme.Product/src/Acme.Product.Desktop/wwwroot/src/app.js) 仍然是单点总控。
2. 弹层 / 向导体系碎片化，存在多套关闭和销毁语义。
3. 生命周期和状态流高度依赖 `window.*`、`document.*` 与全局事件。

如果不先收敛这三类问题，`handEyeCalibWizard.js` 这类问题后续仍会反复出现。
