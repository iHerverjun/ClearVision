## 二、🔴 高风险项（P0级）

### R1：形状匹配性能瓶颈

#### 风险描述
360度旋转搜索 × 多尺度 × 金字塔层级 = 可能超过1秒，无法满足实时检测要求（<200ms）

#### 影响评估
- **影响程度**：高（阻塞工业应用）
- **发生概率**：高（W3-4实现时很可能遇到）
- **影响范围**：形状匹配算子、所有依赖匹配的下游算子

#### 根本原因
- 暴力搜索复杂度：O(W × H × A × S × L)
  - W×H：搜索图尺寸
  - A：角度步数（360度/1度=360）
  - S：尺度数
  - L：金字塔层数

#### 缓解策略

**策略1：角度步长自适应**（优先级：P0）
```markdown
实施时间：W3-4实现时同步
负责人：算法开发者
具体措施：
- 粗层：10度步长（36次旋转）
- 中层：5度步长（候选区域±20度范围）
- 细层：1度步长（候选区域±5度范围）
预期效果：角度搜索加速10x
```

**策略2：早期终止机制**（优先级：P0）
```markdown
实施时间：W3-4实现时同步
具体措施：
- 设置分数阈值（如0.3）
- 当前位置分数<阈值，直接跳过
- 粗层快速筛选，减少细层计算量
预期效果：搜索空间减少50-70%
```

**策略3：搜索区域限制**（优先级：P1）
```markdown
实施时间：W4优化阶段
具体措施：
- 基于先验知识限制搜索区域
- 支持用户指定ROI
- 基于上一帧结果预测（跟踪模式）
预期效果：搜索空间减少80%
```

**策略4：GPU加速**（优先级：P2，W12考虑）
```markdown
实施时间：W12或第4阶段
具体措施：
- 使用CUDA/OpenCL加速梯度计算
- 并行化角度搜索
- 考虑使用TensorRT优化
预期效果：加速5-10x
```

#### 监控指标
- 执行时间：目标<200ms，告警阈值>250ms
- 搜索空间大小：记录实际搜索的位置数
- 早期终止率：记录跳过的位置百分比

#### 应急预案
如果W4结束时性能仍不达标：
1. 降低角度精度（2度步长）
2. 减少金字塔层数（3层→2层）
3. 限制搜索区域为必选参数

### R2：点云内存消耗

#### 风险描述
100万点 × (xyz + rgb + normal) = 48MB/帧，多相机场景可能OOM（Out of Memory）

#### 影响评估
- **影响程度**：高（系统崩溃）
- **发生概率**：中（W5-8实现3D算子时）
- **影响范围**：所有3D点云算子

#### 根本原因
- 点云数据量大（1M点 = 48MB）
- 多相机同时处理（4相机 = 192MB）
- 中间结果未及时释放
- 缺少内存管理机制

#### 缓解策略

**策略1：流式处理架构**（优先级：P0）
```markdown
实施时间：W5点云数据结构设计时
具体措施：
- 实现分块加载（chunk-based loading）
- 每次处理10万点，处理完释放
- 使用迭代器模式
预期效果：内存峰值降低80%
```

**策略2：内存池管理**（优先级：P1）
```markdown
实施时间：W5-6
具体措施：
- 预分配固定大小内存池
- 复用点云缓冲区
- 避免频繁new/delete
预期效果：减少内存碎片，提升性能
```

**策略3：点云大小上限**（优先级：P0）
```markdown
实施时间：W5点云加载时
具体措施：
- 设置最大点数限制（默认100万）
- 超过限制自动下采样（体素滤波）
- 用户可配置上限
预期效果：防止OOM
```

**策略4：内存监控告警**（优先级：P1）
```markdown
实施时间：W0内存泄漏检测时扩展
具体措施：
- 实时监控进程内存使用
- 超过阈值（1.5GB）告警
- 自动触发GC或下采样
预期效果：提前预警，避免崩溃
```

#### 监控指标
- 进程内存使用：目标<1GB，告警阈值>1.5GB
- 点云大小：记录每次处理的点数
- 内存分配次数：优化目标减少50%

#### 应急预案
如果运行时内存不足：
1. 自动下采样到50万点
2. 释放所有缓存
3. 强制GC
4. 如果仍不足，返回错误提示用户减小点云

### R3：数值稳定性问题

#### 风险描述
Zernike矩、RANSAC、手眼标定对数值精度敏感，可能导致结果不稳定或标定失败

#### 影响评估
- **影响程度**：高（结果不可用）
- **发生概率**：中（W1-2、W7、W11实现时）
- **影响范围**：亚像素检测、点云配准、手眼标定

#### 根本原因
- 浮点数精度损失
- 矩阵病态（条件数过大）
- 迭代算法不收敛
- 输入数据未归一化

#### 缓解策略

**策略1：使用double精度**（优先级：P0）
```markdown
实施时间：所有涉及数值计算的算子
具体措施：
- 中间计算使用double而非float
- 最终结果可转回float
- 关键矩阵运算使用double
预期效果：精度提升，减少累积误差
```

**策略2：条件数检查**（优先级：P0）
```markdown
实施时间：矩阵求逆前
具体措施：
- 计算矩阵条件数（cond(A)）
- 条件数>1e6则告警
- 使用SVD分解代替直接求逆
预期效果：避免病态矩阵导致的失败
```

**策略3：输入数据归一化**（优先级：P0）
```markdown
实施时间：所有拟合和标定算子
具体措施：
- 坐标归一化到[-1, 1]或[0, 1]
- 计算完成后反归一化
- 记录归一化参数
预期效果：提升数值稳定性
```

**策略4：鲁棒估计**（优先级：P1）
```markdown
实施时间：W7 RANSAC、W11手眼标定
具体措施：
- 使用RANSAC剔除离群值
- 使用M-estimator（Huber损失）
- 迭代重加权最小二乘
预期效果：对噪声和离群值鲁棒
```

#### 监控指标
- 条件数：记录关键矩阵的条件数
- 迭代次数：记录优化算法的迭代次数
- 残差：记录拟合/标定的残差

#### 应急预案
如果数值不稳定：
1. 检查输入数据是否异常
2. 增加正则化项
3. 使用更鲁棒的算法（RANSAC）
4. 提示用户改善数据质量

## 三、⚠️ 中风险项（P1级）

### R4：并发与线程安全

#### 风险描述
ONNX Runtime会话、点云处理可能有共享状态，多线程环境下崩溃或结果错误

#### 影响评估
- **影响程度**：中（部分场景受影响）
- **发生概率**：中（多线程使用时）
- **影响范围**：深度学习算子、点云算子

#### 缓解策略

**策略1：明确标注线程安全性**（优先级：P0）
```markdown
实施时间：所有算子实现时
具体措施：
- 文档注释标明是否线程安全
- 示例：[ThreadSafe] 或 [NotThreadSafe]
- 在算子文档中说明
预期效果：用户明确知道如何使用
```

**策略2：使用线程本地存储**（优先级：P1）
```markdown
实施时间：W9深度学习、W5-8点云算子
具体措施：
- 使用ThreadLocal<T>存储会话
- 每个线程独立的ONNX会话
- 避免共享可变状态
预期效果：线程安全
```

**策略3：关键区域加锁**（优先级：P1）
```markdown
实施时间：必要时
具体措施：
- 使用lock或Mutex保护共享资源
- 尽量减小锁粒度
- 避免死锁（锁顺序一致）
预期效果：保证正确性
```

**策略4：并发测试**（优先级：P1）
```markdown
实施时间：W0或W1
具体措施：
- 编写多线程测试用例
- 使用Parallel.For测试
- 压力测试（100线程并发）
预期效果：提前发现并发问题
```

#### 监控指标
- 并发测试通过率：目标100%
- 线程安全标注覆盖率：目标100%

### R5：模型部署一致性

#### 风险描述
ONNX模型训练时的预处理与部署时不一致，导致推理精度下降

#### 影响评估
- **影响程度**：中（精度下降）
- **发生概率**：中（W9-10深度学习集成时）
- **影响范围**：所有深度学习算子

#### 缓解策略

**策略1：模型验证管道**（优先级：P0）
```markdown
实施时间：W9-0（新增任务）
具体措施：
- 实现预处理验证工具
- 对比训练框架和ONNX Runtime输出
- 记录预处理步骤（均值、标准差、尺寸）
预期效果：确保一致性
```

**策略2：模型元数据**（优先级：P0）
```markdown
实施时间：W9模型集成时
具体措施：
- 在模型文件中嵌入预处理参数
- 或使用配置文件（model_config.json）
- 自动读取并应用
预期效果：避免人为错误
```

**策略3：端到端测试**（优先级：P1）
```markdown
实施时间：W9-10
具体措施：
- 使用训练集样本测试
- 对比训练框架和部署结果
- 误差<1%则通过
预期效果：验证部署正确性
```

#### 监控指标
- 模型验证通过率：目标100%
- 推理精度：与训练框架误差<1%

## 四、🟢 低风险项（P2级）

### R7：单线程开发瓶颈

#### 风险描述
假设单人顺序开发，无法并行，进度可能延迟

#### 缓解策略
- 任务原子化，便于多人协作
- 明确模块接口，减少依赖
- 使用Git分支策略（feature branch）

---

### R8：技术债务累积

#### 风险描述
"先跑通，再优化"可能导致性能问题遗留

#### 缓解策略
- 每周五代码审查
- W0建立性能基线，持续监控
- 预留20%时间用于重构

## 六、应急响应流程

### 6.1 风险触发条件

当以下情况发生时，触发应急响应：
1. 性能超出预算>50%
2. 内存使用>2GB
3. 测试覆盖率<60%
4. 关键算子失败率>10%
5. 进度延迟>2周

    ↓

    ↓
实施方案
    ↓

    ↓

    ↓

## 七、经验教训库

### 7.1 W1-1成功经验

**任务**：灰度重心法亚像素边缘检测
**预计时间**：2小时
**实际时间**：1.5小时
**成功因素**：
- 算法简单明确
- OpenCV有参考实现
- 测试数据充分
- AI Prompt质量高

**可复用经验**：
- 简单算法优先
- 充分的测试数据是关键
- 详细的AI上下文注释很有效

### 7.2 2026-03-18 补充经验

- 阶段完成报告提交时，必须同步更新阶段 TODO、总审计和风险状态；否则很容易出现“代码已交付，文档还显示未开始”的假阻塞。
- 阶段验收需要明确区分“代码仓口径”和“部署/现场口径”，避免把真实模型、真实硬件、现场节拍混入主线闭环判断。
- 对历史残留失败要标注“是否阻断当前阶段签收”，避免旧性能基线噪声反向污染新阶段结论。

### 8.1 风险评估矩阵

**影响程度评分**：
- 高（3分）：阻塞项目、系统崩溃、数据丢失
- 中（2分）：功能受限、性能下降、用户体验差
- 低（1分）：轻微影响、可绕过

**发生概率评分**：
- 高（3分）：>50%概率发生
- 中（2分）：20-50%概率发生
- 低（1分）：<20%概率发生

**风险等级 = 影响程度 × 发生概率**：
- 高风险：≥6分
- 中风险：3-5分
- 低风险：≤2分
