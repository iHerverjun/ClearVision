# 算法深度审计报告 - 第 3 批：特征匹配、形状检测与 ROI

## 审计概述
- **审计算子数量**：7 个
- **涉及范围**：模板匹配 (NCC/SQDiff)、形状匹配 (旋转/缩放不变)、几何拟合 (圆/直线/椭圆)、圆测量 (Hough)、轮廓检测 (FindContours)、图像拼接 (特征匹配 ORB / 手动裁剪拼接)、ROI 管理器。
- **审计基准**：NotebookLM 工业机器视觉原理最佳实践（高精度定位与测量环节）。

---

## 算子详细审计

### 1. `TemplateMatchOperator` (模板匹配)
- **科学性**：4/5。封装了 OpenCV 标准的 `MatchTemplate`，提供了 NCC (CCoeffNormed) 和 SQDiff 等选项。
- **性能效率**：4/5。底层高度优化。
- **参数合理性**：4/5。
- **鲁棒性**：3/5。不支持**带角度寻优（旋转不变性）**和**多尺度寻优（缩放不变性）**。这在自动化上下料等角度随机的工业场景中非常受限。
- **工程质量**：5/5。输出了详细的位置、全套 Score。
- **改进建议**：当前仅适用于完全平移的目标查找。应该在文档或 UI 提示该算子无旋转不变性。

### 2. `ShapeMatchingOperator` (形状匹配 - 旋转/缩放)
- **科学性**：4/5。通过灰度图旋转模板 + 多角度遍历 + CCoeffNormed NCC 计算 + NMS (非极大值抑制) 实现，是一种经典的穷举旋转匹配。
- **性能效率**：2/5。
  - **严重缺陷**：虽然使用了 `Parallel.ForEach` 进行角度遍历，但在 C# 图层多次 `RotateImage(tmplGray, angle)` 产生大量内存分配。
  - 更致命的是，参数中定义了 `NumLevels` (金字塔层数)，但**代码中完全没有运用图像金字塔(Image Pyramid)**！在原图尺寸上穷举 360 度甚至细分到 0.1 度，会造成灾难性的 CPU 阻塞。
- **参数合理性**：3/5。暴露了参数却没有实现逻辑（NumLevels）。
- **鲁棒性**：4/5。
- **工程质量**：2/5。存在“画大饼”逻辑缺失。
- **改进建议**：
  1. (P0) 必须补充实现图像金字塔（在顶层小图上粗略找角度，在底层大图上精细化角度）。
  2. 修复 `NumLevels` 参数不起作用的 Bug。

### 3. `GeometricFittingOperator` (几何拟合)
- **科学性**：5/5。直线 FitLine (L2，内置最小二乘/Huber)、椭圆 FitEllipse，以及自己手写的严格最小二乘圆拟合 (Kasa 方法)。科学性极佳。
- **性能效率**：5/5。
- **参数合理性**：5/5。
- **鲁棒性**：4/5。在边缘点存在严重离群值（如高亮反光导致轮廓破缺）时，最小二乘（L2）会被拉偏。建议在高级选项中增加基于 RANSAC 或 Huber/Tukey 权重的抗噪拟合选项。
- **工程质量**：5/5。

### 4. `CircleMeasurementOperator` (圆测量)
- **科学性**：4/5。使用 `HoughCircles` (基于梯度的霍夫变换)。
- **性能效率**：4/5。
- **参数合理性**：5/5。暴露了 dp (累加器分辨率)、minDist、param1 (Canny 高阈值)、param2 (同心度阈值)。
- **鲁棒性**：4/5。增加的圆度计算 (Circularity) 非常好，可以过滤非标准圆。不过内部的圆度计算是重新截取 ROI 并跑了一遍 `Canny + FindContours`，逻辑有些重。
- **工程质量**：4/5。
- **改进建议**：霍夫圆对整图的计算量极大。建议最佳实践是：先用 ROI 管理器定位大致区域，再接圆测量。

### 5. `FindContoursOperator` (轮廓检测)
- **科学性**：5/5。OpenCV 的 `FindContours` 封装，支持层级 (External/List/Tree) 和近似 (Simple/TC89)。
- **性能效率**：5/5。
- **参数合理性**：4/5。内置了全局阈值二值化。这其实略显耦合。
- **鲁棒性**：5/5。
- **工程质量**：5/5。

### 6. `ImageStitchingOperator` (图像拼接)
- **科学性**：4/5。支持基于 ORB 特征的自动拼接和手动计算 Overlap 拼接的缝合。
- **性能效率**：4/5。
- **参数合理性**：4/5。
- **鲁棒性**：3/5。ORB 在无明显纹理（例如纯色背景的金属件）上极易失败导致 RANSAC 找不出 Homography 矩阵。
- **工程质量**：4/5。
- **改进建议**：工业图像拼接经常涉及到镜头畸变，应建议用户先执行相机标定去畸变，再做拼接，否则拼缝处会发生扭曲断裂。

### 7. `RoiManagerOperator` (ROI 管理器)
- **科学性**：5/5。支持矩形、外接圆形、多边形（通过 JSON 传入端点序列 `FillPoly` 构建 Mask）。设计得非常贴合工业流。
- **性能效率**：5/5。
- **参数合理性**：5/5。
- **鲁棒性**：4/5。提供了 Crop 和 Mask 两种运算模式，对后续算子的衔接极其友好。
- **工程质量**：5/5。

## 结论与行动项 (Action Items)

1. **核心 Bug 修复（P0）**：`ShapeMatchingOperator` 参数中定义了金字塔层数（NumLevels），但核心代码全是在原图尺寸上进行穷举旋转匹配。不使用金字塔加速会导致高分辨率旋转匹配耗时数秒到几十秒，必须补充金字塔匹配实现。
2. **算法增强（P2）**：在 `GeometricFittingOperator`（尤其是圆拟合和直线拟合）中，增加基于 RANSAC 的剔除离群点机制，以应对工业实际中反光、缺损带来的轮廓干扰。
3. **架构解耦建议（P3）**：`CircleMeasurementOperator` 自己实现了灰度转二值、求轮廓计算圆度，如果能复用前端的 `FindContoursOperator`，则模块化更好。
