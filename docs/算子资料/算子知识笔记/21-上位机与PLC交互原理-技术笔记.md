# 上位机与 PLC 交互原理技术笔记

> 这篇笔记的目标不是把你训练成电气工程师，而是让你在视觉软件岗位面试里，把 `上位机 - PLC - 相机 - IO - 执行机构 - HMI` 这条链讲顺，并且能回扣到 ClearVision。

---

## 1. 一句话先理解这个专题

上位机和 PLC 的交互，本质上是在做三件事：

1. **控制命令下发**
2. **设备状态同步**
3. **检测结果回写**

PLC 负责稳定、确定、毫秒级的现场控制；上位机负责复杂计算、视觉算法、界面交互、历史记录和配方管理。工业视觉通常就站在两者之间。

---

## 2. 先把角色分工讲清楚

### 2.1 工业现场里常见的几个角色

| 角色 | 最擅长的事 | 不擅长的事 | 面试里怎么讲 |
|------|------------|------------|--------------|
| **PLC** | IO 控制、节拍控制、状态机、联锁、安全逻辑 | 复杂图像算法、复杂 UI、长链路数据分析 | “PLC 更像现场控制中枢” |
| **上位机** | 视觉算法、AI、配方管理、界面、日志、报表、历史追溯 | 替代 PLC 做实时硬件联锁 | “上位机更像大脑和操作台” |
| **HMI/工位屏** | 简单按钮、状态显示、参数输入 | 大规模数据处理、复杂建模 | “HMI 更偏现场操作界面” |
| **相机** | 采集图像 | 做复杂业务判断 | “相机是感知入口，不是决策中心” |
| **传感器 / IO 模块** | 检测到位、光电、限位、急停等 | 复杂业务逻辑 | “负责原始现场信号” |
| **执行机构** | 气缸、机械手、分拣器、报警灯等动作执行 | 自己判断业务逻辑 | “只负责执行，不负责思考” |

### 2.2 一个更容易记忆的比喻

- **PLC** 像一个非常稳定的现场班组长，擅长按固定节拍管动作、管安全、管联锁。
- **上位机** 像一个能看图、会分析、会记账、还能让人操作的工程站。
- **相机** 像眼睛。
- **执行机构** 像手。

所以工业视觉系统不是“谁替代谁”，而是：

> `PLC 负责稳，上位机负责聪明，视觉负责看懂。`

---

## 3. 一个典型的现场链路长什么样

```text
         [HMI / 工位屏]
                |
                v
[上位机 / 视觉软件] <------> [PLC] <------> [IO / 执行机构]
        |                       |
        v                       v
      [相机]                 [传感器]
```

更具体一点：

```text
工件到位
   |
   v
PLC 收到到位信号
   |
   +--> 触发相机 / 允许拍照
   |
   v
上位机拿到图像并做视觉检测
   |
   v
上位机把 OK / NG / 测量值 回写给 PLC
   |
   v
PLC 决定 放行 / 剔除 / 报警 / 停机
```

这就是面试里最常见的“视觉系统怎么和现场设备协同”的最小闭环。

## 4. PLC 为什么重要，但又不直接替代视觉

### 4.1 PLC 的强项

- 输入输出稳定
- 扫描周期固定
- 梯形图 / 状态机适合做联锁
- 和现场电气设备集成成熟
- 对“动作一定要发生”和“动作绝不能乱发生”更有保障

### 4.2 PLC 的弱项

- 不适合跑复杂图像处理
- 不适合管理大量图像和历史结果
- 不适合做复杂的 UI 交互
- 不适合承担 LLM、模板管理、场景规则和资源配置

### 4.3 面试里最稳的表达

> PLC 不是拿来替代视觉算法的，它更像一个稳定的现场控制器。上位机负责看图、算结果、做人机交互，PLC 负责把这些结果变成可靠的现场动作。

## 5. 位、字、寄存器这些概念到底是什么

### 5.1 为什么一定要懂这几个词

因为上位机和 PLC 交互时，最终经常不是在传“对象”或“接口”，而是在读写：

- 一个触发位
- 一个完成位
- 一个报警位
- 一个寄存器里的数值
- 一段配方参数区

### 5.2 最小概念表

| 概念 | 可以怎么理解 | 常见用途 |
|------|--------------|----------|
| **Bit / 位** | 只有 0 或 1 | 触发、完成、报警、允许 |
| **Word / 字** | 一般是 16 位 | 数值、状态码、计数 |
| **DWord / 双字** | 一般是 32 位 | 更大的整数、时间戳、组合值 |
| **寄存器** | PLC 里存数据的地址单元 | 读写状态、参数、结果 |
| **地址** | 具体去哪里读写 | 比如 `D100`、`M0`、`DB10.DBW2` |

### 5.3 一个直觉化例子

```text
D100 = 触发命令
D101 = 检测结果
D102 = 缺陷数量
D200~D210 = 配方参数区
```

上位机如果读到 `D100 = 1`，就知道 PLC 在发起本次检测。

上位机如果把 `D101 = 1` 写回去，PLC 可能解释为 OK；写 `0` 可能解释为 NG。  
这个编码规则不是协议天然规定的，而是双方约定的“现场语义”。

## 6. PLC 扫描周期为什么会影响视觉触发

### 6.1 PLC 的基本工作方式

PLC 不是像 Web 服务那样“请求来了再处理一次”，它更像一个不停循环的控制器：

```text
读取输入
  ->
执行逻辑
  ->
刷新输出
  ->
再次读取输入
  ->
再次执行逻辑
```

这整个循环就叫一个**扫描周期**。

### 6.2 为什么这件事和视觉有关系

因为现场触发常常是短脉冲或节拍信号：

- 如果信号太短，可能刚出现就消失
- 如果上位机轮询太慢，可能错过触发
- 如果相机和 PLC 的时序没对齐，可能拍到上一拍或下一拍

### 6.3 一个典型误区

很多人会把“PLC 发了触发”理解成“上位机一定立刻准确收到并正确消费”。  
实际上中间还隔着：

- 通信延迟
- 轮询间隔
- 软件线程调度
- 相机曝光时间
- 本次检测是否还没处理完

所以面试里要知道：

> 视觉触发不是一个单点动作，而是一段时序。

## 7. 上位机通常怎么和 PLC 交互

### 7.1 最常见的是轮询 + 请求响应

工业现场里，上位机经常会做这种事：

1. 周期性读 PLC 某个地址
2. 看值有没有变化
3. 满足条件就开始拍照 / 检测 / 回写

这就是**轮询**。

而大多数工业协议本质上是：

> 上位机发请求，PLC 回响应。

它不像 Web 前端那样天然强调事件推送，也不像消息队列那样默认有复杂订阅模型。

### 7.2 为什么工业现场很爱轮询

- 逻辑简单
- 设备兼容性高
- 地址化读写容易标准化
- 对 PLC 和上位机都容易落地
- 在很多老设备上，轮询本来就是最现实的方式

### 7.3 轮询的代价

- 太慢会增加响应延迟
- 太快会增加网络和 PLC 压力
- 如果没有“沿触发 / 完成位 / 清零位”，容易重复消费同一信号

## 8. 软件触发、硬件触发、PLC 触发到底是什么关系

### 8.1 软件触发

上位机主动让相机拍一张。

适合：

- 调试
- 离线演示
- 节拍要求不高
- 上位机主导流程

### 8.2 硬件触发

相机通过外部电信号触发拍照，常见触发源就是 PLC、光电传感器、编码器等。

适合：

- 节拍要求高
- 到位就拍
- 对时序稳定性要求高

### 8.3 PLC 触发

“PLC 触发”不是一个和前两者平行的新类别，而是要再问一句：

> PLC 触发的到底是谁？

它有两种常见落法：

1. **PLC 直接给相机外部触发信号**  
   这在本质上属于**硬件触发**
2. **PLC 通过通信告诉上位机‘现在该检测了’**  
   这在本质上是**PLC 发状态，上位机再软件触发**

### 8.4 面试里怎么说最稳

> 软件触发和硬件触发描述的是“相机怎么被拍”；PLC 触发描述的是“谁在发起这次节拍”。PLC 既可以通过 IO 直接硬触发相机，也可以通过通信让上位机去执行软件触发。

## 9. 一个最经典的交互时序

### 9.1 PLC 触发拍照 -> 上位机检测 -> 回写 OK/NG

```text
PLC: 工件到位
  |
  +--> 置 Trigger=1
  |
  v
上位机: 读到 Trigger=1
  |
  +--> 采图
  +--> 视觉检测
  +--> 得到 OK / NG
  |
  v
上位机: 回写 ResultCode / Done
  |
  v
PLC: 读取结果并执行分拣 / 放行 / 报警
```

### 9.2 如果没有“完成握手”会怎样

常见问题包括：

- PLC 还没来得及读结果，上位机已经把值覆盖了
- 上位机连续两次读到同一个 Trigger，重复检测
- PLC 没清 Trigger，上位机不断重复消费同一拍

所以现场系统很少只靠一个“触发位”，而是常见这几类信号组合：

- Trigger
- Busy
- Done
- Ack
- ResultCode

### 10.2 一个很适合面试的说法

> 在 ClearVision 里，我不会把 PLC 通信孤立地讲成“读写一个寄存器”这么简单，而会把它放进完整链路里讲：相机怎么被触发、视觉链怎么跑、结果怎么判定、OK/NG 怎么回写、异常怎么兜底。

### 10.3 要注意的边界

- 不要把通信成功说成业务成功
- 不要把上位机说成实时控制替代 PLC
- 不要把 HMI、上位机、PLC 混成一个东西
- 不要把“读到触发位”说成“系统时序已经可靠”

## 11. 面试高频追问怎么接

### 11.1 “上位机和 PLC 的交互本质是什么？”

稳定答法：

> 本质上是控制命令、状态同步和结果回写。PLC 擅长现场控制，上位机擅长视觉算法和人机交互，工业视觉系统通常需要两者协同，而不是让谁替代谁。

### 11.2 “为什么 PLC 不直接做视觉？”

稳定答法：

> 因为 PLC 更适合确定性的 IO 和状态机控制，不适合承担复杂图像处理、模型推理、界面展示和历史记录。视觉一般由上位机完成，PLC 接收结果后决定现场动作。

### 11.3 “PLC 为什么常用轮询和寄存器？”

稳定答法：

> 因为工业现场首先看重简单、兼容、稳定。轮询和寄存器读写实现成本低、厂商支持广、调试方式成熟，所以在很多现场仍然是主流方案。

## 12. 常见误区

### 误区 1：把 PLC 触发理解成一个单一概念

实际上要分清：

- PLC 触发相机
- PLC 触发上位机
- PLC 只负责给“允许检测”状态

### 误区 2：把“读到了值”当成“时序可靠”

现场要看的是完整闭环，不是某一次读写成功。

### 误区 3：把上位机当 PLC 用

上位机可以协调流程，但不该承担本来属于 PLC 的强实时联锁职责。

### 误区 4：只谈协议，不谈业务语义

协议只负责把值传过去；`1` 代表 OK 还是 NG、何时清零、谁先确认，这些是现场约定。

---

## 13. 一句话总结

> 上位机和 PLC 的交互，不是“谁更强谁替代谁”，而是把 `稳定控制` 和 `复杂计算` 各自放在最合适的位置，再通过触发、状态、结果回写把整条工业视觉链闭起来。
