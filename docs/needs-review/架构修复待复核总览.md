- Phase 1 的 Scoped 生命周期问题已被 `Coordinator + Worker` 结构替代。

## 问题与修复映射

### 1. 实时检测状态生命周期错误

诊断结论是：`InspectionService` 以 Scoped 生命周期持有跨请求运行态，导致启动/停止和并发状态不可靠。

当前修复路径已经清晰：

- 用 `InspectionRuntimeCoordinator` 持有进程级状态。
- 用 `InspectionWorker` 管理实际运行任务。
- 将 `InspectionService` 收敛为 API 门面，不再承担状态容器角色。

这部分从方案、实施总结和闭环审查三个来源看，都已经有稳定一致的完成信号。

### 2. 命令面与事件面的通信分裂

诊断文档将问题定义为：

- 命令走 HTTP。
- 事件结果仍依赖宿主侧 WebMessage。
- 多客户端监听和外部集成都受限。

现有落地结果显示：

- 已有 `InspectionEventEndpoints` 和 `InMemoryInspectionEventBus`。
- 已支持 `Last-Event-ID` 回放和 30 秒心跳。
- 前端已切换为 `SSE 优先，WebMessage 回退` 的双栈模式。

所以 Phase 2 也更像是“已交付待确认最终验收”，而不是“仍未开始”。

### 3. 预览子图与 LLM 闭环能力

这里是文档信号最矛盾的部分：

- `架构修复实施总结.md` 仍把 Phase 3 / 4 写成“待审批项”。
- 但闭环完成报告里，Phase 3 已被审查为完成，Phase 4 工程实现也被判定为 95% 完成。

这说明 `实施总结.md` 的状态已经滞后，不适合继续作为主状态源。

## 当前最值得复核的点

### A. Phase 3 是否已闭环

从 [架构修复方案审查报告](../closed/阶段性完成报告/架构修复方案审查报告.md) 看，以下能力已经给出明确完成信号：

- `DebugOptions.BreakAtOperatorId`
- `POST /api/flows/preview-node`
- 前端 `previewNode()` 接入
- 对应自动化测试

如果仓库当前代码与该报告一致，这一项应直接从 `needs-review` 移出。

### B. Phase 4 的边界是否需要拆项

Phase 4 里实际上混了两类事情：

- 工程实现：指标采集、分析、调参服务、接口。
- 验证闭环：真实样本、统计检验、实验设计。

前者多数已被完成报告覆盖，后者则明显依赖环境与数据。

更合理的做法是：

- 将“工程实现完成”归入 `closed/`。
- 将“真实样本验证 / 科学实验”单独拆成 `active/` 任务。

### C. 这组文档的最终归属

如果不再把“真实样本验证”算作架构修复主任务的一部分，那么架构修复本身其实已经接近可以闭环。

1. 以闭环完成报告为准，复核 Phase 3 的代码证据是否仍在当前主链。

3. 将当前这份总览移入 `closed/`，原 `拆分稿` 进入 `reference/` 或继续保留作追溯。
