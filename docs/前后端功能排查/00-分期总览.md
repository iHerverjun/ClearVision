- 拆解方式：主代理整合 3 个 `GPT-5.4` 子代理结果后收敛生成

## 分期原则

1. 先恢复真实可用性，再扩展新能力。
2. 先统一前后端契约，再推进页面与数据流改造。
3. 对“假按钮 / 假配置 / 假成功提示”优先去伪存真，避免继续误导用户。
4. 后端已实现但未暴露的能力，先做“保留 / 接入 / 下沉”决策，再安排开发。
5. 同一业务能力只保留一套主入口和一套主协议，避免双轨漂移继续扩大。

## 问题映射

| 报告分类 | 代表问题 | 默认落点 |
| --- | --- | --- |
| A. 明确接线 / 契约错误 | `/api/api/*`、分页参数不一致、登出只清前端、AI 历史回放不稳定 | Phase 0 ~ Phase 1 |
| B. 前端伪实现 / 假按钮 / 假配置 | 假连接状态、演示值、假清空、假自动保存、假配置页 | Phase 1 ~ Phase 2 |
| C. 后端已实现但前端没有入口 | 自助改密、自动调参、分析报告、图片缓存、设置重置 | Phase 1 ~ Phase 2 |
| D. 双轨实现漂移风险 | 标定两套入口、两套协议并存 | Phase 1 ~ Phase 3 |

## 推荐执行顺序

1. 完成 Phase 0，确保核心链路全部可调用、可验证。
2. 完成 Phase 1，把“页面显示什么”和“系统真实做什么”彻底对齐。
3. 完成 Phase 2，决定哪些后端能力正式开放到 UI，哪些保持内部能力。
4. 完成 Phase 3，收敛历史包袱，清理残留入口，固化准入基线。

## 阶段退出标准

### Phase 0

- 不再出现 `/api/api/*` 这类重复前缀请求。
- AI 健康检查、旧标定链路、节点预览链路全部可联调通过。
- 形成一份最小回归检查清单，后续改动默认回归。

### Phase 1

- 结果页分页、登出、AI 历史回放契约全部统一。
- 前端不再伪造关键业务状态或展示演示值冒充真实数据。
- 结果页开始以服务端分析能力为主数据源，至少覆盖基础统计与趋势。

### Phase 2

- 所有“看起来能点”的入口，都能真实执行或被明确禁用。
- 已决定开放的后端能力均有 UI 入口、状态反馈和验收路径。
- 配置页中展示的字段，全部能保存、回显、解释清楚生效条件。

### Phase 3

- 标定只保留一套入口和一套协议。
- 明确淘汰旧容器、旧模块、已废弃页面与多余协议。
- 形成一份前后端功能矩阵，后续新增功能必须先落矩阵再开发。

## 建议协作方式

- 前端：负责入口、文案、状态展示、交互与页面联调。
- 后端：负责契约、DTO、接口语义、配置模型与实际执行能力。
- 联调：负责冒烟清单、验收口径、灰度切换与废弃路径确认。
- 文档：每完成一个阶段，同步更新本目录对应 TODO 文档状态与结论；涉及主入口或主协议变化时，同步更新基线文档。
