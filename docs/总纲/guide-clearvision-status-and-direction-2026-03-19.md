# ClearVision 项目现状与未来方向纲领

> 版本基线：2026-03-19  
> 文档目的：站在当前项目进度节点，对 ClearVision 已实现能力、与初衷的偏差、以及真正可落地的未来方向做一次战略性收敛。  
> 参考依据：`docs/fixbug+未来规划_0319/README.md`、`docs/guides/guide-codebase-deep-dive.md`，并结合当前代码主链核对。

---

## 1. 一句话判断

ClearVision 到今天为止，已经不是一个“AI 拼出来的视觉 Demo”，而是一个**具备完整平台雏形的工业视觉桌面产品骨架**：有画布编排、有执行引擎、有相机与 PLC 接入、有实时检测、有结果分析、有 AI 辅助生成。

但与此同时，项目也出现了一个非常明显的阶段性偏差：

**平台能力的横向扩张速度，明显快于“一个高频场景能否闭环落地”的纵向验证速度。**

换句话说，当前最需要的不是继续把平台做得更大、更全，而是把路线从：

**“154 个算子 + LLM 自由编排 + 工人直接搭流程”**

收敛到：

**“高频场景模板库 + 节点预览反馈 + 少量关键参数调优 + 人机协同闭环”**。

这不是退而求其次，而是更接近你最初“让现场真正能用起来”的目标。

## 2. 当前项目现状

## 2.1 平台层：已经具备较完整的产品骨架

从代码实现看，ClearVision 当前已经具备以下主能力：

- 桌面宿主 + 本地 Web 服务 + Web 前端的产品架构。
- 工程管理、流程保存、流程画布编辑、节点参数编辑、单次检测、实时检测。
- 相机管理、PLC 通信、结果中心、分析报表、用户与权限、设置中心、标定能力。
- AI 生成流程、会话历史、附件输入、流式进度与结果应用。

这说明项目的“底盘”并不差，反而已经相当完整。  
如果只看平台完成度，ClearVision 已经超过了很多内部工具型项目。

## 2.2 算子层：能力很强，但复杂度也已经明显抬升

当前仓库中的算子目录已经从之前文档里的 `118` 增长到 `154`。  
而且能力范围已经不只是传统 2D 检测，实际上还覆盖了：

- 图像预处理、分割、Blob、测量、OCR、条码、模板匹配、深度学习。
- 标定、坐标变换、结果输出、脚本、通信。
- 更进一步的 3D 点云、纹理、分块、拼接等扩展能力。

这说明“算法底座”在继续变强，但也带来了一个直接后果：

**用户认知负担已经明显大于现场可用性收益。**

尤其对“工人自主搭流程”这个目标来说，154 个算子不是优势，反而是门槛。

## 2.3 运行闭环：已经开始从“能跑”走向“可信”

你在 `0319` 规划里提出的三件关键事，代码层已经出现了明确落地：

- 实时检测运行时从 `InspectionService` 单点内存状态，演进到了 `InspectionRuntimeCoordinator + InspectionWorker` 的进程级协调模式。
- 实时事件链路已经有 `IInspectionEventBus + SSE + WebMessage fallback` 的统一方向。
- 节点预览已经不是单算子孤立预览，而是支持 `POST /api/flows/preview-node` 执行到目标节点并返回最小反馈指标。

这三点非常关键，因为它们决定了你后续做“LLM 半自动调参”有没有土壤。  
没有稳定运行、稳定反馈、稳定预览，再强的模型也只能停留在“连线助手”。

## 2.4 AI 主链：已经不只是 Prompt 拼接，但仍然主要停留在“生成初版流程”

当前 AI 主链已经具备：

- 会话式上下文。
- 多模型配置与连接器。
- 动态裁剪相关算子目录。
- 流式输出。
- 结构化 JSON 校验。
- Dry-Run / 验证兜底。
- 生成后直接应用到画布。

这说明项目已经从“纯聊天”走向了“面向工作流生成的结构化 AI”。

但它的本质仍然是：

**让模型根据算子元数据和大 Prompt，生成一份初版工作流 JSON。**

这条链路在“选算子、连结构”方面是有价值的，但在“把参数调到现场可用”方面仍然偏弱。

## 2.5 模板与调参：能力已经出现，但还没有真正串成一个产品主路径

当前代码里已经能看到三块很重要的半成品能力：

- 有独立的 `FlowTemplateService`，并内置了 8 个模板。
- 有前端 `templateSelector.js`，可手动套用模板。
- 有 `PreviewMetricsAnalyzer`、`AutoTuneService`、`/api/autotune/*` 相关接口。

这说明你真正需要的方向，其实代码已经在“局部自发出现”：

**模板化、指标化、调参化。**

但问题在于，这几块现在还没有被串成默认主流程。  
它们更像是“平台里已经有的若干能力”，而不是“用户真正走的那条半自动化主链”。

## 3. 与最初愿景的对照

## 3.1 已经对齐的部分

从初衷上看，项目并没有走偏到完全失控，反而有几件事是做对了的：

### A. “画布式通用视觉平台”这个底座已经立住了

这一点是成立的。  
当前平台已经能承担多工程、多流程、多设备、多结果管理，不是玩具。

### B. “借助 LLM 降低搭建门槛”这个方向也是对的

问题不在“是否应该引入 LLM”，而在“LLM 应该被放在哪一层发挥作用”。

### C. “先做公司内部多产线、多场景复用”这个定位是合理的

ClearVision 目前最适合的身份，不是对外商用平台，而是内部生产场景工具平台。  
这个定位是健康的，也更符合资源现实。

## 3.2 已经出现偏差的部分

真正的偏差，不是“做了 AI”，而是以下几条：

### A. 把“工人自主搭流程”理解成了“工人面对全量算子库”

这一步其实不现实。  
无论算子名片写得多细，工人真正缺的都不是参数说明书，而是：

- 这个场景该从哪一类模板起手。
- 先看哪一个中间节点。
- 哪几个参数可以调。
- 调大调小分别意味着什么。

所以问题不在文档不够厚，而在产品抽象层级不对。

### B. 把 LLM 的职责压到了“直接调算法参数”

这也是一个关键偏差。  
LLM 擅长的是：

- 理解意图。
- 组织结构。
- 提供解释。
- 根据反馈给出下一步建议。

LLM 不擅长的是：

- 在连续参数空间里盲调数值。
- 单靠文字推断图像分布。
- 在没有执行反馈的情况下把视觉参数一次猜准。

所以它不应该单兵作战，而应该配合：

- 模板先验。
- 预览结果。
- 最小指标。
- 规则调参器 / 启发式调参器。

### C. 平台在“还没证明一个场景能闭环”之前，扩得太宽了

现在项目里已经有很多强能力，但其中一部分在当前阶段属于“能力储备”，不是“价值主线”：

- 大量高级算子。
- 3D 点云能力。
- 各类扩展通信与脚本能力。
- 较重的分析、报表、兼容链路。

这些东西不能说没价值，但它们显然不是当前最影响你愿景成败的那一环。

### D. 你投入很多精力整理了算子名片，但 AI 主链并没有完整消费这套资产

这一点很重要。

当前 AI 生成主链主要依赖的是**运行时算子元数据**和 `PromptBuilder` 动态裁剪后的算子目录，而不是 `docs/operators/*.md` 这套厚文档本身。

这意味着：

- 你“做文档”的方向不是错。
- 但文档资产还没有真正转化为产品主链里的“场景知识”与“调参知识”。

换句话说，现在缺的不是更多算子说明，而是**把说明升级为场景规则、模板先验和反馈策略**。

## 4. 当前项目最真实的定位

如果今天要重新给 ClearVision 定位，我认为最准确的表述不是：

**“对标 Vision Master 的通用低代码视觉平台”**

而是：

**“面向公司内部高频视觉场景的模板化检测平台，正在从人工搭流程，过渡到 AI 辅助生成初版流程与半自动调参。”**

这句话比“通用平台”更克制，但更真，也更有落地路径。

## 5. 真正的未来方向纲领

## 5.1 总方向

未来 1 个阶段，ClearVision 不应该继续把重点放在：

- 扩算子数量。
- 强推 LLM 自由编排全库。
- 让工人直接面对画布和全量算子。

而应该明确转向：

**高频场景模板库 + 节点级预览反馈 + 白名单参数调优 + LLM 协同助手。**

## 5.2 产品目标重定义

真正应该追求的，不是“一句话 + 一张图就自动生成可投产工程”，而是：

### 对工人

让工人不需要理解 154 个算子。  
工人只需要：

- 选择场景模板。
- 上传样图或连接相机。
- 看几个关键中间结果。
- 反馈“检不出来 / 误报多 / 通讯没打通 / 结果不稳定”。

### 对工程师

让工程师负责：

- 沉淀模板。
- 限定候选算子集。
- 定义关键参数范围。
- 定义场景目标指标与失败诊断规则。

### 对 LLM

让 LLM 的职责从“全能流程编排者”调整为：

- 场景理解器。
- 模板选择器。
- 初版流程生成器。
- 反馈解释器。
- 参数微调建议助手。

而不是“脱离反馈的全量算子自由拼装引擎”。

## 5.3 真正该形成的主流程

未来主流程应该是下面这条：

1. 用户选择或描述场景。
2. 系统先从场景模板库里选最近模板，而不是直接面对全量算子。
3. LLM 在收缩后的候选算子集内生成初版流程。
4. 系统执行并返回目标节点预览图与最小指标。
5. 用户基于现场结果给出反馈。
6. LLM 或规则调参器只修改少量关键参数。
7. 当结果可用后，由工程师将其固化为模板配方。

这条路径的关键词不是“自动生成”，而是：

**模板起步、反馈驱动、参数收敛、经验沉淀。**

## 5.4 平台该保留什么、收缩什么

### 必须保留并继续强化的

- 画布引擎与流程执行引擎。
- 相机、PLC、结果输出等产线落地能力。
- 节点预览、调试缓存、最小指标输出。
- 参数推荐、自动调参、错误诊断。
- 模板库、项目管理、会话历史。

### 应该收缩到“工程师模式”或暂不作为主卖点的

- 全量算子直接暴露。
- 3D / 点云 / 高阶实验性算子在默认路径中的存在感。
- 多套并行 AI 编排链路。
- 与当前高频场景无关的能力扩张。

### 当前阶段明确不该再加码的

- 继续扩算子库本身。
- 追求多场景一键泛化。
- 把工人培训目标继续建立在“理解算子”而不是“理解模板”上。

## 5.5 数据资产的重点不再是“算子说明书”，而是“场景包”

未来真正有复利的数据资产，不是单个算子的长文档，而是“场景包”：

- 场景描述。
- 10 到 50 张样图。
- 黄金模板流程。
- 收缩后的候选算子集。
- 关键参数白名单。
- 最小指标与阈值。
- 常见失败模式及调整建议。

也就是说，知识组织方式要从：

**算子中心**

转向：

**场景中心。**

## 6. 对当前代码路线的具体判断

## 6.1 目前最有价值的积累

从“未来还能不能做成”来看，当前最值钱的不是 154 个算子本身，而是以下几块：

- `InspectionRuntimeCoordinator + InspectionWorker` 让实时执行开始可信。
- `InspectionEventEndpoints + SSE/WebMessage` 让反馈链路开始统一。
- `PreviewNodeEndpoints` 让中间节点调试开始可视化。
- `PreviewMetricsAnalyzer + AutoTuneService` 让“反馈驱动调参”第一次有了代码入口。
- `AiFlowGenerationService + PromptBuilder + Validator + DryRun` 让 AI 生成从聊天走向结构化工作流。
- `FlowTemplateService + templateSelector` 让模板化方向已经出现雏形。

这些才是你真正的“未来筹码”。

## 6.2 目前最明显的结构性断点

当前最该优先补上的不是新能力，而是把以下几件事接起来：

### 断点 1：模板库和 AI 主链是分开的

现在模板是模板，AI 是 AI。  
模板没有真正成为 AI 的第一起点。

### 断点 2：自动调参已经有后端入口，但还没成为用户主路径

它存在，但还不是用户默认可感知的产品能力。

### 断点 3：工人视角还没有真正建立“简化模式”

虽然系统有角色概念，但算子层并没有形成真正的“工人模式 / 工程师模式”分层。

### 断点 4：场景知识还没有压过算子知识

现在系统更像“知道很多算子”，而不是“非常懂几个高频场景”。

## 7. 建议的下一阶段路线

## 7.1 第一优先级：把“包装箱两条白色包装带检测”做成真正闭环样板

这不是一个普通 demo，而应该是未来路线的第一个“黄金样板”。

这个样板至少要具备：

- 固定场景模板。
- 收缩后的候选算子集。
- 节点预览。
- `blobCount / areaStats / binaryRatio / errorMessage` 最小反馈。
- 2 到 3 轮参数微调。
- 模板封装与再次复用。

## 7.2 第二优先级：把模板库变成主入口，而不是辅助入口

建议未来 AI 入口改成：

- 先判定属于哪个场景族。
- 再读取对应模板与候选算子集。
- 最后才生成初版流程。

不是先让 LLM 面对全库，再靠 Prompt 勉强约束。

## 7.3 第三优先级：做真正的“工人模式”

工人模式应该具备：

- 看不到全量算子库。
- 只能选模板。
- 只能调有限几个关键参数。
- 能看到关键中间图和简单解释。
- 能把现场反馈转交给系统。

工程师模式才继续保留完整画布与全量算子。

## 7.4 第四优先级：清理并收敛 AI 主链

建议未来只保留一条明确的主链：

- 模板选择。
- 候选算子裁剪。
- 初版生成。
- 预览反馈。
- 参数微调。
- 模板沉淀。

旧的兼容链、重复的 AI 编排入口、没有进入产品主路径的设计，应逐步降权。

---

## 8. 30/60/90 天建议

## 30 天内

- 打通 1 个高频场景的模板闭环。
- 让 AI 生成链默认基于模板和候选算子集工作。
- 把节点预览和最小指标稳定接入调参流程。
- 把自动调参先作为工程师调试工具开放。

## 60 天内

- 做出 3 到 5 个真正可复用的黄金模板。
- 建立工人模式和工程师模式。
- 为每个高频场景形成“样图 + 模板 + 关键参数 + 常见故障”的场景包。

## 90 天内

- 在真实产线找 1 到 2 个点位试运行。
- 衡量模板复用率、调参时间、人工介入次数、误报漏报情况。
- 再决定是否继续往“更强的通用平台”扩张。

---

## 9. 最终结论

ClearVision 现在最可贵的地方，不是它已经做了很多，而是它已经做出来的那部分里，**恰好包含了通往正确方向所需要的关键部件**：

- 可执行的平台底座。
- 可视化画布。
- 可调试的节点预览。
- 可收缩的 AI 生成。
- 可沉淀的模板机制。
- 可演进的调参能力。

所以这个项目不是“应该推倒重来”，而是“应该强力收敛主线”。

真正的未来方向，不是继续追逐“全量通用 + 一键生成 + 工人自由搭图”的想象。

真正的未来方向是：

**先把 1 个高频场景做到可闭环，再把它沉淀成模板；再把 3 到 5 个高频场景做到可复用；最后再谈通用平台。**

也就是说，下一阶段 ClearVision 的核心命题应该从：

**“平台能做多少事”**

改成：

**“现场是否能借助模板、预览和反馈，把一个场景稳定做成。”**
