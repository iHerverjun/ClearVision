## 第二问：数据验证与AI边界

**面试官追问 2**:

> 你的回答更具体了，但我听到了几个需要验证的断言。
>
> 1. **"AI自动根据模板构建流程"** — 这个"自动"的边界在哪里？是 LLM 直接输出一个可运行的 JSON 流程，还是输出一个草稿需要人再确认？比如工人说"我要检测端子线序"，AI 是直接把 8 个算子连好线、参数配好，还是只把相关算子列出来让人自己拖？
> 2. **"多轮对话微调参数"** — 这里的参数微调，是 LLM 直接改算子里的数值（比如把阈值从 0.5 改成 0.7），还是 LLM 说"建议把阈值调低"，然后工人手动改？如果是 LLM 直接改，它怎么知道现场图像是偏亮还是偏暗？
> 3. **"5个工作日 vs 2周，提效超过100%"** — 这个数据是真实项目统计的，还是你根据工作量估算的？VisionMaster 的 2 周包含了哪些环节（采图、标定、调参、验证）？你们的 5 天又包含哪些？如果去掉"熟悉 VisionMaster 的时间"，纯开发时间的对比是多少？
> 4. **（你刚才回避的问题）模板是谁做的？** 如果是你们预置的，现在有多少个场景模板？如果是现场工程师配的，工程师需要培训多久才能配出一个合格的模板？

**候选人回答**:

> LLM会直接输出一个可以运行的JSON流程，比如工人说"我要检测端子线序"，AI 会直接把 8 个算子连好线、参数配好。当然，在每个算子细致的调参方面，就像如面试官您所提到的， LLM 直接改，它很难知道现场图像是偏亮还是偏暗，所以在这一步工人可以手动去调节参数，也可以在与LLM的对话框里跟他描述画面情况。如果LLM模型是支持图片传入的多模态LLM，本软件还支持直接发送图片。
>
> 提效问题是我估计的。在开发本项目过程中我更多还是关注的自己项目的提高，暂时还未全面跟竞品对比，而且我们主要的优势还是在LLM辅助视觉工程构建这一块。
>
> 模板是由我本人预置的，目前着重强调了目标检测这一高频场景的调优，其他场景还在推进中。由于该项目完全由我本人单人开发，所以模板这一块还在规划补充中。

## 第三问：技术边界与工作量合理性

**面试官追问 3**:

> 诚实度加分——承认数据是估算、模板在规划中、单人开发，这比硬撑好。
>
> 但我听到了几个技术边界需要澄清，以及一个工作量合理性的疑问。
>
> 1. **LLM 直接输出"可运行的 JSON 流程"** — 这个"可运行"的边界需要厘清。比如端子线序检测，LLM 怎么知道：
>
>    - 现场相机 ID 是什么？
>    - 深度学习模型的 `.onnx` 文件路径在哪里？
>    - 检测区域的 ROI 坐标是多少？
>      是 LLM 猜一个默认值（比如用 "camera_0"、默认 ROI 全图），还是你们有某种**现场配置预注入**机制？
> 2. **多模态 LLM 支持图片传入** — 如果工人可以传图给 LLM，让 LLM 看画面亮度，那**为什么不让 LLM 直接给出具体的阈值参数**（比如"建议阈值从 0.5 调到 0.3"），还要工人手动去属性面板改？这一步自动化的卡点在哪里？
> 3. **单人开发 50 万行代码 + 160 个算子** — 这个工作量我需要帮你算笔账：
>
>    - 50 万行代码，按每天 500 行有效代码（含调试、重构），需要 **1000 个工作日 ≈ 4 年**
>    - 即使每天 1000 行，也需要 **500 个工作日 ≈ 2 年**
>    - 这还没算测试、文档、现场调试
>      **你实际开发了多久？** 这 50 万行是全手写，还是包含了生成的代码（比如 EF Core Migration、前端脚手架）？160 个算子全是你一个人写的吗？

**候选人回答**:

> 首先回答您的第一个问题：本软件采用的是预先在设置页面配置全局相机，图像采集算子挂载相机ID的模式，LLM不需要给定具体的相机ID，这一块工作量较小且难度较低，由工人完成即可。 .onnx 文件路径同样也支持手动指定位置。检测区域的 ROI 坐标目前做的是支持鼠标手动滑动选取。
>
> 关于您的第二个问题：这一步其实是没卡的，LLM当然可以直接给出具体的阈值，我之所以说让工人手动去属性面板调，是因为受限于目前LLM能力局限（哪怕是最强的claude opus4.6），本项目的思路从来都不是完全去人工化，而是强调AI作为copilot辅助落地，人工作为审计和把关。
>
> 关于您的第三个问题，这也是我需要跟您小小自夸一下的一个点（笑）。此项目确确实实是我一个人从零开始独立构建的，而且从立项到现在总共开发时间约为2个月。这是因为我非常关注AI以及科技圈相关的动态和新闻，本人也一直在学习和研究AI最前沿的技术。在开发本软件的过程中，我通过AI助手完成了大量复杂工作的提速构建，然后通过人工复审和算法调优，这才以一个在早些年完全不敢相信的速度把一个项目从零做到现在这个规模。我本人有着较强的接受能力和学习能力，相信在目前AI重塑整个计算机领域的大背景下能够创造更大价值。

## 第六问：LLM选算子的准确性保障

**面试官追问 6**:

> 你最后的回答**展示了真实的工程迭代过程**——从全量Prompt到模板收缩，这是符合行业演进规律的。但我必须指出一个**关键的概念混淆**：
>
> 你说"第二次就可以跑通**格式校验**"——但格式校验（JSON Schema合法、算子ID存在、端口类型匹配）**不等于业务正确**。
>
> - **格式校验通过**：TemplateMatch → BlobDetection → LogicGate 这条链能跑起来，不报错
> - **业务正确**：这条链能检测出线序，而不是把端子当成Blob数个数
>
> **你们现在怎么保证业务正确？** 还是只能靠人工肉眼判断"这流程看起来不对"？
>
> 如果工人不懂视觉算法（你假设的目标用户），他看到LogicGate输出TRUE/FALSE，怎么知道这是"检测线序"还是"数端子个数"？

**候选人最终回答**:

> 是这样的，在最初立项的时候，我尝试为我每个算子设计一个名片（现在这些.md格式的算子名片还存在我的项目目录里，里面记载了每个算子的名字、实现方法、参数和端口类型），然后通过预先给定与视觉检测相关的Prompt（在prompt里规定好比如第一个算子要从图像采集开始……如果提到PLC要选用对应的通讯算子等）来规范LLM模型的输出（参考了目前AI行业惯用的SKILL.md的范式），这样我实测下来比较前沿的LLM模型（如我实测过的DeepseekV3.2、Gpt5.4等），都能够根据我预先设定的规范生成出可用工程流。
>
> 如果LLM选错路径，项目会：
> ①先做一轮语法检测，避免LLM模型生成出不符合格式的Json链
> ②如果通过了这一轮语法检测，即生成了格式上可用的工程后，由人工来评估是否采用。
>
> 当然我知道这存在不足（比如LLM模型在选自具体的实现时可能选取不同路径，其中包含了你说的B路径：TemplateMatch → BlobDetection → LogicGate（错，传统方法搞不定线序）），所以后面我改用了之前提到过很多次的【模板化思路】，即我会规定好一大类的工程AI应该采用哪些算子（可以理解为精细化的Prompt），当LLM模型识别到用户自然语言里出现的关键词时（如缺陷检测），加载那一类特定的模板提示词，收缩LLM模型可选择模型的范围，避免LLM模型在150+全量算子库里盲选。模板思路的最终形态计划还会增加参数的推荐等，尽量减少多轮对话出现的频率。
>
> 最后一个问题，如果LLM第一次生成就是错的，我实测一般在第二次就可以跑通格式校验。
