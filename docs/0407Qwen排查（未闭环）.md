## 一、严重 Bug

### 1. `401.py:1164-1178` — `while True` 循环实际只执行一次

```python
while True:
    loopCount += 1
    ...
    break  # <-- 第一次循环就 break，永远不会真正循环
```

**问题**：`while True` 循环在第一次迭代末尾无条件 `break`，导致即使用户设置 `RUN_ONCE=False`，也只会执行一次检测，完全失去了循环检测的意义。

**建议**：根据条件决定是否 `break`，而非无条件跳出。

### 2. `AutoTuneService.cs:121-128, 287-304` — OpenCV Mat 内存泄漏

```csharp
Mat? outputImage = null;
if (executionResult.OutputData?.TryGetValue("Image", out var imgObj) == true && imgObj is Mat mat)
{
    outputImage = mat;
}
...
var metrics = _metricsAnalyzer.Analyze(outputImage, ...);
```

**影响范围**：

- `AutoTuneInFlowAsync` 方法（约第 287 行）

**建议**：使用后调用 `outputImage?.Dispose()`，或使用 `using` 语句。

### 3. `ImageAcquisitionService.cs:391-395` — Mat 双重释放风险

```csharp
using var resizedMat = new Mat();
Cv2.Resize(resultMat, resizedMat, ...);
resultMat.Dispose();
resultMat = resizedMat.Clone();
```

**问题**：`resultMat` 被手动 `Dispose()` 后重新赋值。如果 `Clone()` 抛出异常，`resultMat` 已经被 Dispose，而外层 `catch` 块会再次调用 `resultMat?.Dispose()`，对已释放对象调用 `Dispose()` 可能抛出 `ObjectDisposedException`。

**建议**：重构为安全的模式，避免手动 `Dispose` 后重新赋值。

---

### 4. `CameraManager.cs:138-146` — Dispose 中同步阻塞异步操作导致死锁

```csharp
public void Dispose()
{
    if (!_disposed)
    {
        DisconnectAllAsync().Wait();  // <-- 同步阻塞异步操作
        _disposed = true;
    }
    GC.SuppressFinalize(this);
}
```

**问题**：在 `Dispose` 中使用 `.Wait()` 同步阻塞异步方法，在存在 `SynchronizationContext` 的环境下（如 WPF/WinForms UI 线程）可能导致死锁。

**建议**：实现 `IAsyncDisposable`，或在 `Dispose` 中使用 `.ConfigureAwait(false).GetAwaiter().GetResult()`。

## 二、中等严重度

### 5. `ConnectionPoolManager.cs:44-81` — TOCTOU 竞态条件

```csharp
if (_connections.TryGetValue(key, out var existingConnection) && existingConnection.IsValid)
{
    return (IModbusMaster)existingConnection.Connection;
}
// 创建新连接...
```

**问题**：`TryGetValue` 检查和后续创建新连接之间存在 Time-of-Check to Time-of-Use 竞态条件。两个线程可能同时发现连接不存在/无效，然后都创建新连接，导致资源浪费和连接泄漏。

**建议**：使用 `ConcurrentDictionary.GetOrAdd` 或加锁保护整个检查-创建流程。

---

### 6. `ConnectionPoolManager.cs:246-259` — Dispose 非线程安全

```csharp
public void Dispose()
{
    if (_disposed) return;
    _heartbeatTimer?.Dispose();
    foreach (var connection in _connections.Values)
    {
        connection.Dispose();
    }
    _connections.Clear();
    _disposed = true;
}
```

**问题**：`_disposed` 检查不是原子操作，多线程同时调用 `Dispose` 可能导致重复释放资源。

**建议**：使用 `Interlocked.Exchange(ref _disposed, 1) == 0` 进行原子检查。

### 7. `FlowExecutionService.cs:310` — 线程安全问题

```csharp
var layerResults = await Task.WhenAll(layerTasks);
result.OperatorResults.AddRange(layerResults);
```

**问题**：`FlowExecutionResult.OperatorResults` 是 `List<OperatorExecutionResult>`，不是线程安全集合。如果在同一 `FlowExecutionResult` 实例上并发调用 `ExecuteFlowAsync`，会引发竞态条件。

**建议**：确保每个执行使用独立的 `FlowExecutionResult` 实例，或使用线程安全集合。

### 8. `OpenAiConnector.cs:26-32` — 共享 HttpClient 被修改和释放

```csharp
_httpClient = httpClient ?? new HttpClient();
...
_httpClient.BaseAddress = new Uri(...);
_httpClient.DefaultRequestHeaders.Add("Authorization", ...);
_httpClient.Timeout = _config.Timeout;
```

**问题**：
1. 当外部传入 `httpClient` 时，`Dispose()` 仍会 dispose 它（第 240 行），可能 dispose 了不属于它的 HttpClient
2. 直接修改共享 HttpClient 的 `BaseAddress`、`DefaultRequestHeaders` 和 `Timeout` 会影响所有使用该 client 的代码

**建议**：记录 HttpClient 所有权，仅在自行创建时才 Dispose；使用 `HttpRequestHeaders` 而非 `DefaultRequestHeaders` 设置认证。

### 9. `ConnectionPoolManager.cs:96-144` — 返回的 NetworkStream 存在悬垂引用风险

```csharp
public async Task<NetworkStream> GetOrCreateTcpConnectionAsync(...)
{
    ...
    return stream;  // 返回 NetworkStream
}
```

**问题**：返回的 `NetworkStream` 依赖于连接池内部的 `TcpClient`。如果连接池因健康检查回收了连接，外部持有的 `NetworkStream` 会变成悬垂引用，后续读写会失败。

**建议**：返回包装对象而非裸 `NetworkStream`，或在文档中明确声明生命周期。

---

### 10. `InspectionRuntimeCoordinator.cs:197` — Fire-and-forget 吞异常

```csharp
_ = CleanupSessionAsync(projectId);
```

**问题**：在 `MarkAsFaulted` 和 `MarkAsStopped` 中使用 fire-and-forget 模式。如果 `CleanupSessionAsync` 抛出异常，异常会被静默吞掉，导致资源泄漏或状态不一致。

**建议**：至少添加 `.ContinueWith(t => LogException(t.Exception), TaskContinuationOptions.OnlyOnFaulted)`。

---

### 11. `InspectionRuntimeCoordinator.cs:291` — `_locks` 字典内存泄漏

```csharp
// 注意：不清理 _locks，避免并发问题
```

**问题**：`_locks` 中的 `SemaphoreSlim` 永远不会被清理。长时间运行的服务会持续累积已停止会话的锁对象，造成内存泄漏。

**建议**：在会话彻底结束后（如延迟一段时间后）安全地清理对应的锁条目。
