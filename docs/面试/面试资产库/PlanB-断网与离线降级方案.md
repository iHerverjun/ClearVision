# Plan B：断网与离线降级方案

> 用途：回答“如果外网不能用、API 涨价、项目涉密怎么办？”
> 原则：不是空泛地说“那就先不用 AI”，而是给出当前仓库事实能支撑的降级层次。

---

## 1. 当前最稳的结论

ClearVision 的 Plan B 不应该设计成“没有 LLM 就彻底不可用”，而应该是一个分层降级体系：

1. 模板优先
2. 本地模型优先
3. 规则和人工确认优先
4. 关键场景先保住，非核心场景后退

---

## 2. 当前可依赖的基础

### 2.1 模板侧基础

- `FlowTemplateService` 负责模板加载与持久化
- `PromptBuilder` 已支持 `template-first`
- 线序检测有完整的 scenario package

### 2.2 本地模型侧基础

- `LLMConnectorFactory` 支持：
  - `OpenAI`
  - `AzureOpenAI`
  - `Ollama`
- `OllamaConnector` 是真实存在的本地模型路径，不是 PPT 里的占位概念

### 2.3 无 AI 时仍可保留的系统能力

- 流程编辑器
- 模板选择
- 运行时执行
- 预览调参
- 结果输出

这意味着：AI 不是系统唯一入口。

---

## 3. 降级层次

### 3.1 Level 1：云端 LLM 可用

模式：

- `PromptBuilder + template-first + 云端 LLM + Validator + DryRun + 人工确认`

适用：

- 正常联机场景

### 3.2 Level 2：外网不可用，但本地模型可用

模式：

- `PromptBuilder + template-first + Ollama 本地模型 + Validator + DryRun + 人工确认`

适用：

- 外网受限
- API 成本上升
- 内网可部署本地模型

### 3.3 Level 3：LLM 全不可用，但模板仍可用

模式：

- `Template Selector + 已有场景包 + 手工补参数 + 预览调参 + 运行时执行`

适用：

- 完全断网
- 项目涉密
- 本地模型暂不可部署

### 3.4 Level 4：只保核心 20% 场景

模式：

- 优先维护高频模板和少数关键算子
- 放弃低频、长尾、强依赖自由生成的体验

适用：

- 资源紧张
- 团队规模小
- 需要先保业务连续性

---

## 4. 当前最现实的降级顺序

1. 先保 `wire-sequence-terminal` 这类高频模板场景
2. 再保模板选择、参数编辑、单算子预览、结果输出
3. 再考虑是否启用 `Ollama` 本地模型
4. 最后才谈恢复更自由的自然语言生成

---

## 5. 哪些功能在降级后仍能保住

| 功能 | 云端 LLM | 本地模型 | 无 LLM 纯模板 |
|---|---|---|---|
| 模板选择 | 可以 | 可以 | 可以 |
| 流程编辑器手动调整 | 可以 | 可以 | 可以 |
| 预览调参 | 可以 | 可以 | 可以 |
| 运行时执行 | 可以 | 可以 | 可以 |
| 自然语言自由生成 | 可以 | 部分可以 | 不可以 |
| 缺资源自动提示 | 可以 | 可以 | 需要模板和人工配合 |

---

## 6. 面试时最稳妥的回答

> 我不会把 ClearVision 设计成一个“没有云端 LLM 就完全失效”的系统。当前最现实的 Plan B 有三层：第一层是模板优先，即使 AI 不可用也能通过已有模板和场景包继续搭建；第二层是本地模型优先，仓库里已经有 `Ollama` 接入路径；第三层是在极端情况下只保住高频模板场景和核心算子，把自由生成体验降级掉。  
>  
> 这样做的核心思想是，AI 是提速器，不是系统唯一依赖。

---

## 7. 现在还需要继续补什么

- 某个具体场景的“无 LLM 演示录屏”
- 本地模型下的实际效果记录
- 纯模板模式下的标准操作手册

这三样一旦补齐，Plan B 会更像“可执行方案”，而不只是设计思路。
