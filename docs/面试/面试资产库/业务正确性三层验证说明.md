# 业务正确性三层验证说明

> 用途：专门回答“格式正确不等于业务正确”。
> 背景：这是 [面试复盘-2026-03-24.md](../面试复盘-2026-03-24.md) 里最需要纠正的概念混淆之一。

---

## 1. 先说结论

对 ClearVision 这类“LLM 生成流程”的系统来说，至少要把下面三层分开：

1. `Schema 合法`
2. `链路可运行`
3. `业务结果正确`

如果只说到前两层，就不能宣称“流程已经对了”。

---

## 2. 三层分别是什么意思

| 层级 | 问题本质 | 典型检查项 | 通过后说明什么 | 通过后仍然不能说明什么 |
|---|---|---|---|---|
| 第一层：Schema 合法 | JSON/结构有没有错 | JSON 可解析、算子类型存在、端口名存在、参数格式合法 | 这份流程草案能被系统识别 | 不代表连得通，更不代表业务对 |
| 第二层：链路可运行 | 流程在运行时能不能执行 | 端口类型兼容、无环路、资源准备齐全、DryRun/执行不崩 | 这条链路至少能跑完 | 不代表输出结果符合业务语义 |
| 第三层：业务结果正确 | 输出是否满足业务验收 | 样本、预期结果、NG 原因、验收标准、人工复核 | 这条流程在特定场景下“确实做对了事” | 仍不等于全部场景通用 |

---

## 3. 仓库里分别对应到哪里

### 3.1 第一层：Schema 合法

核心证据：

- `Acme.Product/src/Acme.Product.Infrastructure/AI/AiFlowValidator.cs`
- `Acme.Product/src/Acme.Product.Infrastructure/AI/PromptBuilder.cs`

当前可讲点：

- `AiFlowValidator` 会校验算子类型、端口存在、类型兼容、无环路、输入端口不重复、参数合法性。
- `PromptBuilder` 约束 AI 输出为一个合法 JSON 对象，并支持 `recommendedTemplate / pendingParameters / missingResources` 这些辅助字段。

### 3.2 第二层：链路可运行

核心证据：

- `Acme.Product/src/Acme.Product.Infrastructure/AI/DryRun/DryRunService.cs`
- `Acme.Product/src/Acme.Product.Infrastructure/Services/FlowExecutionService.cs`
- `Acme.Product/src/Acme.Product.Infrastructure/AI/DryRun/DryRunStubRegistry.cs`

当前可讲点：

- DryRun 能在不依赖完整外设的情况下预演流程。
- 运行时有独立的 `FlowExecutionService`、`InspectionWorker` 和 `InspectionRuntimeCoordinator` 主干。
- 这说明流程不只是“长得像 JSON”，而是能进入执行链路。

### 3.3 第三层：业务结果正确

核心证据：

- `线序检测/scenario-package-wire-sequence/README.md`
- `线序检测/scenario-package-wire-sequence/manifest.json`
- `线序检测/scenario-package-wire-sequence/template/terminal-wire-sequence.flow.template.json`
- `线序检测/scenario-package-wire-sequence/rules/sequence-rule.v1.json`
- `线序检测/线序检测结果可观测性与NMS收口执行记录-2026-03-23.md`

当前可讲点：

- 线序检测不是只看 JSON 能跑通，而是要满足 `Wire_Black -> Wire_Blue` 的顺序契约。
- 业务正确性依赖：
  - 正确标签
  - 正确计数
  - 正确排序方向
  - 正确阈值 owner
  - 正确 ROI 约束

## 4. 用线序检测举一个最容易懂的例子

### 4.1 只过了第一层

- AI 输出的 JSON 能解析。
- 算子类型没写错。
- 连线字段也都完整。

这时最多只能说：

> “流程草案结构上可被系统接受。”

### 4.2 过了第二层

- Validator 通过。
- DryRun 或运行时能把流程跑起来。
- 没有因为端口、参数或资源直接报错。

这时最多只能说：

> “流程可以运行。”

### 4.3 真正到了第三层

还必须额外满足：

- 模型标签与规则文件一致。
- ROI 没把目标裁掉，也没把无关区域引入。
- `BoxNms` 真正承担阈值与 NMS 职责。
- `DetectionSequenceJudge` 判断出的顺序和业务要求一致。
- 结果输出能解释为什么 OK / NG，而不是只有一个真假值。

这时才可以说：

> “这条流程在端子线序场景里业务上是正确的。”

---

## 5. 面试里最稳的回答模板

> 我现在会把验证分成三层。第一层是 Schema 合法，解决的是 JSON 能不能被系统接收；第二层是链路可运行，解决的是这条流程能不能在 Validator、DryRun 和运行时里真正跑起来；第三层才是业务结果正确，也就是它在真实样本和验收规则下有没有把事情做对。  
>  
> 所以我不会把“AI 生成了合法 JSON”直接说成“业务已经正确”。像线序检测这种场景，真正的业务正确还要看标签、ROI、阈值职责、排序规则和最终输出解释是否一致。

---

## 6. 常见错误说法

- `格式都对了，所以流程就是对的。`
- `DryRun 跑通了，所以现场就能用。`
- `结果输出 TRUE/FALSE 了，所以业务一定对。`

这些都应该替换成：

- `结构上通过`
- `执行链路上通过`
- `业务验收上还需要样本与规则验证`
