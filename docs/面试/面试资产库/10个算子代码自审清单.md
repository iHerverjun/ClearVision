# 10 个算子代码自审清单

> 用途：回答“你怎么做代码审查”“你会重点看哪些风险”。
> 核心：不是逐行背代码，而是形成一套共性风险扫描框架。

---

## 1. 先看 6 类共性风险

1. `参数校验不一致`
2. `输入输出契约模糊`
3. `图像通道/数据类型处理不统一`
4. `路径/资源/配置来源不清`
5. `数值边界与默认值不稳`
6. `图像对象、Mat、序列化结果的资源释放与导出不清`

---

## 2. 10 个算子自审表

| 算子 | 先审什么 | 最容易出什么问题 |
|---|---|---|
| `Threshold` | 阈值范围、`UseOtsu` 与 `Type` 的关系 | 参数组合不一致，导致行为与预期不符 |
| `CannyEdge` | `Threshold1/Threshold2`、自动阈值、输出契约 | 调参逻辑不透明，输出边缘图难解释 |
| `Morphology` | `Operation`、核大小、迭代次数 | 开闭/腐蚀/膨胀混用导致结果偏差 |
| `BlobDetection` | 面积/形状筛选、孔洞处理、输出特征字典 | 输出契约太复杂，边界样本容易失真 |
| `FindContours` | 轮廓层级、输入前景质量、输出类型 | 把轮廓当成业务语义本身 |
| `TemplateMatch` | 模板输入格式、分数阈值、匹配数 | 只在单目标静态场景可用，却被说成通用方案 |
| `CameraCalibration` | 棋盘格参数、输入图像质量、输出 JSON | 结果能输出，但误差分析不充分 |
| `Undistort` | 标定数据依赖、图像尺寸适配 | 标定文件对了但输入图尺寸不匹配 |
| `WidthMeasurement` | `MeasureMode`、采样数、线输入来源 | 自动/手动两种模式边界没讲清 |
| `CircleMeasurement` | 边缘点质量、拟合稳定性、输出单位 | 噪声下拟合漂移，但仍被当成精确值 |

## 3. 审代码时要问的固定问题

### 3.1 参数层

- 默认值和文档写的是不是一致？
- 最小值、最大值、枚举值有没有真正约束？
- 自动模式和手动模式是不是都能解释清楚？

### 3.2 输入输出层

- 输入端口真的必填吗？
- 输出里哪些字段给 UI 看，哪些字段给下游算子连？
- 输出对象能不能稳定序列化，不把 `Mat` 直接泄漏到上层？

### 3.3 图像与数据层

- 是灰度图、彩色图还是二值图？算子自己有没有假设？
- 通道数不一致时会不会直接行为漂移？
- 图像和字节数组的转换有没有重复拷贝或释放不清？

### 3.4 业务层

- 这个算子输出的是“中间几何结果”，还是“业务判定结果”？
- 这个算子是不是被误拿来承担了不该由它承担的职责？
- 如果结果错了，最先应该怀疑它自己，还是怀疑它的上游输入质量？

## 4. 最适合在面试里举的三个例子

### 4.1 `BlobDetection`

要点：

- 它不是单纯“有结果就行”，而是输出契约和特征字典都很重。
- 所以审它时要优先看特征值解释是否稳定。

### 4.2 `CannyEdge`

要点：

- 不是只看能不能出边，而是要看阈值策略有没有可解释性。
- 自动阈值时，最好能暴露实际使用的阈值。

### 4.3 `CameraCalibration`

要点：

- 不是只看能不能输出 `CalibrationData`。
- 更关键的是输入图质量、棋盘参数和后续使用链路是否一致。

---

## 5. 面试里怎么说更像工程师

> 我做代码自审时，不会只盯着实现对不对，而是会先看六类共性风险：参数校验、输入输出契约、图像通道和数据类型、资源配置来源、数值边界以及资源释放和序列化。这样我在看不同算子时，就不会变成“逐行读代码”，而是能快速判断它最容易在哪个层面出问题。
