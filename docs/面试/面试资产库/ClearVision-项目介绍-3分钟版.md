# ClearVision 项目介绍 3 分钟版

> 用途：当面试官说“展开讲讲你这个项目”时使用。
> 目标：3 分钟内讲清“为什么做、做成什么、自己负责什么、当前边界在哪里”。

---

## 1. 标准口播版

ClearVision 这个项目，我现在不会把它包装成一个已经成熟商用的视觉平台，而是把它定义成一个围绕视觉流程编排、模板复用和 LLM 辅助生成的原型系统。它想解决的问题是，传统视觉项目里“流程怎么搭、参数怎么配、场景怎么复用”这件事对经验依赖很强，迁移和复用成本也高。

所以我做的主线，不是单纯堆更多算子，而是想把“自然语言需求到视觉流程”的过程做成一条可收敛的链路。具体来说，用户先描述需求，系统优先判断是否命中高频模板场景，比如端子线序检测；如果命中，就先用模板把可选空间收窄，再让 LLM 生成结构化流程 JSON，同时带出推荐模板、待确认参数和缺失资源。生成之后，不是直接就算业务正确，而是还要经过 Validator 做结构校验、经过 DryRun 做预演和覆盖收集，最后保留人工确认环节，再进入运行时执行。

我个人最主要的贡献，是这条链路的设计与收敛。包括 Prompt 结构、模板优先策略、待确认参数和资源缺口提示、Validator、DryRun、消息桥接，以及线序检测场景包这一类可以复用的试点资产。仓库里像 `PromptBuilder`、`AiFlowGenerationService`、`AiFlowValidator`、`DryRunService`、`GenerateFlowMessageHandler`、`FlowTemplateService` 这些地方，都能对应到这条主线。

从可自证程度来说，我现在最强的不是“我对所有算子都同样深入”，而是“我把 AI 生成视觉流程这件事，从自由生成逐步收敛成模板优先、可校验、可预演、可人工审计的链路”。线序检测场景是目前最完整的一条证据链，因为它已经有模板、规则、manifest、FAQ、版本记录，以及一次真实的 NMS 职责收口和诊断输出补强。

当然，这个项目也有边界。我现在不会说它已经对所有场景都稳定落地，也不会把没有证据的效率数字直接当成果。我更愿意诚实地说：它已经跑通核心链路，沉淀出部分可验证场景，但距离成熟产品还有继续补证据、补工程化和补业务验证的空间。

## 2. 三分钟版本的结构提示

按下面顺序说，最不容易失真：

1. 先讲它解决什么问题。
2. 再讲完整链路，而不是先报功能列表。
3. 再讲自己负责的主线。
4. 最后主动承认边界，反而更可信。
