# 黄金场景包 02：模板定位与宽度测量

> 成熟度：受控演示场景。
> 证据等级：算子测试为主，不包装成“已落地完整场景包”。

---

## 1. 这个场景的定位

这个场景不应该包装成“当前最成熟业务场景”，它更适合作为一个受控演示包，用来回答：

- 传统视觉里模板匹配是干什么的？
- 为什么定位之后还需要测量？
- 如何把“匹配”和“测量”拆成两个责任明确的步骤？

当前最可靠的证据来自：

- `Acme.Product/tests/Acme.Product.Tests/Operators/TemplateMatchOperatorTests.cs`
- `Acme.Product/tests/Acme.Product.Tests/Operators/WidthMeasurementOperatorTests.cs`

---

## 2. 输入样本

### 2.1 当前可公开样本形式

- `TemplateMatchOperatorTests` 中使用 `120 x 120` 黑底图，目标区域是一个白色方块。
- `WidthMeasurementOperatorTests` 中使用受控输入图像，并手动给出两条测量线：
  - `Line1 = x=20`
  - `Line2 = x=40`

### 2.2 这代表什么

- 模板匹配这部分证明了：在受控场景下，算子能以模板图和原图完成匹配，并输出可用结果图。
- 宽度测量这部分证明了：在受控场景下，手动给入测量线后，算子能稳定给出约 `20 px` 的宽度结果。

---

## 3. 场景目标

这个场景最适合讲成“两步法”：

1. `TemplateMatch` 先做定位或存在性确认
2. `WidthMeasurement` 再对受控区域做宽度测量

它的价值在于让面试官看到：

- 我知道定位和测量不是一回事
- 我知道什么时候该用模板匹配，什么时候该转到几何测量

---

## 4. 建议演示骨架

### 4.1 一句话版

先找到目标，再沿已知结构方向做宽度测量，最后输出匹配分数和宽度值。

### 4.2 建议演示流程 JSON 骨架

下面这个 JSON 更适合被理解为“面试演示骨架”，不是仓库内现成场景包导出文件：

```json
{
  "operators": [
    { "tempId": "op_1", "operatorType": "ImageAcquisition", "displayName": "图像采集" },
    {
      "tempId": "op_2",
      "operatorType": "TemplateMatching",
      "displayName": "模板匹配",
      "parameters": {
        "Method": "CCoeffNormed",
        "Threshold": "0.8",
        "MaxMatches": "1"
      }
    },
    {
      "tempId": "op_3",
      "operatorType": "WidthMeasurement",
      "displayName": "宽度测量",
      "parameters": {
        "MeasureMode": "ManualLines",
        "NumSamples": "20"
      }
    },
    {
      "tempId": "op_4",
      "operatorType": "ResultOutput",
      "displayName": "结果输出",
      "parameters": {
        "Format": "JSON",
        "SaveToFile": "false"
      }
    }
  ]
}
```

### 4.3 边界必须讲清

- `TemplateMatch` 的测试证据主要证明“模板输入 + 输出图像可用”，不是完整复杂场景鲁棒性证明。
- `WidthMeasurement` 当前最公开可证的方式是 `ManualLines`，说明这个场景更像受控测量 demo，不该夸大成 fully automatic production flow。

---

## 5. 关键参数

| 参数 | 当前建议值 | 说明 |
|---|---|---|
| `TemplateMatch.Method` | `CCoeffNormed` | 当前默认相似度口径 |
| `TemplateMatch.Threshold` | `0.8` | 先保证匹配可信度，再谈多目标 |
| `TemplateMatch.MaxMatches` | `1` | 演示场景先控制成单目标 |
| `WidthMeasurement.MeasureMode` | `ManualLines` | 当前最稳的公开证据路径 |
| `WidthMeasurement.NumSamples` | `20` | 来自现有测试用例 |

---

## 6. 预期输出

最适合展示的输出包括：

- `TemplateMatch.Position`
- `TemplateMatch.Score`
- `TemplateMatch.IsMatch`
- `WidthMeasurement.Width`

当前受控测试里可直接讲的结果是：

- 宽度结果约为 `20 px`
- 验收范围是 `19.0 ~ 21.0`

---

## 7. 验收标准

1. 模板匹配成功，且返回可用结果图。
2. 匹配位置与模板区域一致，不出现明显漂移。
3. 宽度测量结果稳定落在 `19.0 ~ 21.0`。
4. 当模板缺失时，`TemplateMatch` 能返回失败，而不是无意义成功。

---

## 8. 搭建耗时与调参轮次

### 8.1 当前可公开状态

- `搭建耗时`：暂无统一计时记录，不能对外报“几分钟搭好”。
- `调参轮次`：当前公开证据主要来自测试，不是完整现场调参日志。

### 8.2 面试建议口径

> 这个场景我会明确把它定义成受控演示包。它的作用不是证明我已经把模板定位和尺寸测量做成了成熟业务方案，而是证明我能把“定位”和“测量”这两类传统视觉能力拆开讲清楚，并且已有最小可验证证据。
