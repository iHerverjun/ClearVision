## 2. 场景目标

目标不是“随便检测几根线”，而是在固定业务契约下判断顺序是否正确。

当前默认业务契约是：

- 场景键：`wire-sequence-terminal`
- 期望顺序：`Wire_Black -> Wire_Blue`
- 期望数量：`2`
- 当前排序方向：`TopToBottom`

## 4. 流程骨架

### 4.1 一句话版

先全图检测候选框，再用 ROI 过滤，再由 `BoxNms` 完成唯一业务去重，最后用 `DetectionSequenceJudge` 判断顺序并输出结构化诊断。

### 4.2 当前模板骨架

`ImageAcquisition -> DeepLearning -> BoxFilter(FilterMode=Region) -> BoxNms -> DetectionSequenceJudge -> ResultOutput`

### 4.3 当前模板 JSON 里的关键部分

```json
{
  "scenarioKey": "wire-sequence-terminal",
  "expectedSequence": ["Wire_Black", "Wire_Blue"],
  "requiredResources": ["DeepLearning.ModelPath", "DeepLearning.LabelsPath"],
  "tunableParameters": ["BoxNms.ScoreThreshold", "BoxNms.IouThreshold"]
}
```

以及关键运行参数：

```json
{
  "DeepLearning": {
    "Confidence": "0.05",
    "EnableInternalNms": "false",
    "TargetClasses": "Wire_Black,Wire_Blue"
  },
  "BoxNms": {
    "ScoreThreshold": "0.25",
    "IouThreshold": "0.45"
  },
  "DetectionSequenceJudge": {
    "ExpectedLabels": "Wire_Black,Wire_Blue",
    "ExpectedCount": "2",
    "MinConfidence": "0.0"
  }
}
```

## 6. 关键参数

| 参数 | 当前职责 | 为什么重要 |
|---|---|---|
| `DeepLearning.ModelPath` | 模型资源路径 | 没它流程无法形成真实检测能力 |
| `DeepLearning.LabelsPath` | 标签资源路径 | 标签不对，业务判断会整体失真 |
| `BoxFilter.RegionX/Y/W/H` | ROI 过滤范围 | 决定是否把无关目标引入排序链路 |
| `BoxNms.ScoreThreshold` | 置信度筛选阈值 | 现场优先调参项之一 |
| `BoxNms.IouThreshold` | 去重阈值 | 现场优先调参项之一 |
| `DetectionSequenceJudge.ExpectedLabels` | 业务标签契约 | 决定判断逻辑到底在比什么 |
| `DetectionSequenceJudge.ExpectedCount` | 业务数量契约 | 解决“检到了但数量不对” |

## 8. 预期输出

当前更符合业务验收的输出，不是只有一句文本，而是同时输出：

- 预览图像
- `BoxNms.Diagnostics`
- `DetectionSequenceJudge.Diagnostics`
- `DetectionSequenceJudge.Message`

这使得一次运行能回答：

- 是没检出？
- 是重复框过多？
- 是顺序错了？
- 还是阈值太严 / 太松？

---

## 9. 验收标准

至少应满足下面几条：

1. 检测到的标签集合与规则文件一致。
2. 排序结果满足 `Wire_Black -> Wire_Blue`。
3. 输出数量满足 `ExpectedCount = 2`。
4. `BoxNms` 诊断可解释保留与抑制结果。
5. `DetectionSequenceJudge` 能说明 mismatch 的原因，而不是只给真假。
