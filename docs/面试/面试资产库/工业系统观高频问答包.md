# 工业系统观高频问答包

> 用途：补足“视觉软件岗”在工业现场系统观上的短板，让你在追问 `上位机 / PLC / 协议 / 时序 / 回写` 时不发虚。  
> 使用原则：始终回扣 ClearVision，不把自己包装成纯自动化工程师。

---

## 1. 上位机和 PLC 到底怎么交互

### 1 分钟版

上位机和 PLC 的交互，本质上是控制命令、状态同步和结果回写。PLC 更擅长现场 IO、节拍和联锁控制，上位机更擅长视觉算法、界面、配方和历史记录。在 ClearVision 里，我会把这件事落到 `ImageAcquisition + ResultJudgment + 通信算子` 这条闭环上，而不是只说“读写一个寄存器”。

### 3 分钟版

如果从工业系统视角看，PLC 更像现场控制中枢，它负责工件到位、动作联锁、剔除、报警这些确定性控制；上位机负责视觉检测、参数配置、界面和结果展示。两者交互通常就是三类信息：第一类是 PLC 给上位机的触发或状态，第二类是上位机回给 PLC 的 OK/NG 或测量值，第三类是上位机和 PLC 之间的配方、状态码或完成确认。  
在 ClearVision 里，我会把这条链讲成：相机由 `ImageAcquisition` 承接，视觉链处理图像后由 `ResultJudgment` 先把业务结果收口，再通过 `MitsubishiMcCommunication`、`ModbusCommunication` 或 `TcpCommunication` 去做回写。这样我强调的不是“会连”，而是“我知道这条链在现场怎么闭环”。

### 追问风险点

- 不要把 PLC 说成视觉算法执行主体
- 不要把上位机和 HMI 混成同一个概念
- 不要只说“读写寄存器”，要讲完整闭环

---

## 2. 为什么 PLC 不直接做视觉

### 1 分钟版

因为 PLC 更适合稳定控制和状态机，不适合复杂图像处理、模型推理、界面和历史记录。视觉一般放在上位机做，PLC 接收结果后执行现场动作，这样分工更合理。

### 3 分钟版

PLC 的优势是确定性和稳定性，比如 IO 控制、联锁、安全逻辑、节拍控制，这些都非常适合 PLC。但图像处理和 AI 推理的特点完全不同，它们需要处理大对象数据、算法链、模型资源、可视化界面和历史结果管理，这些更适合在上位机完成。  
所以我在面试里不会把 PLC 说成“也能做视觉所以都能做”，而是会强调：工业视觉系统里最好的方式是 PLC 负责稳，上位机负责聪明。在 ClearVision 里也是这个思路，视觉链和 LLM 编排在上位机侧，PLC 通信算子只是闭环的一部分，不是项目的全部。

### 追问风险点

- 不要贬低 PLC，重点是“分工不同”
- 不要说成“PLC 完全不能处理任何计算”
- 要把话题拉回“为什么视觉软件岗位需要懂协同边界”

## 3. PLC 为什么常用轮询和寄存器，不像 Web 那样强调事件推送

### 1 分钟版

因为工业现场首先看重简单、稳定、兼容和易调试。轮询和寄存器读写虽然看起来朴素，但在很多设备和 PLC 上都是最成熟、最容易交付的方式。

### 3 分钟版

工业协议和 Web 协议的设计目标不一样。Web 更强调面向用户和服务的交互体验，而工业现场更强调设备兼容、现场可维护、状态清晰和长期稳定。轮询加寄存器读写的好处是语义简单、厂商支持广、现场人员熟悉，调试也容易用现成工具和地址表来做。  
但我也不会把轮询说得过于理想化，因为它会带来时序问题，比如轮询太慢导致响应延迟、轮询太快给 PLC 增压、只靠一个触发位容易重复消费。所以更稳的说法是：轮询是常见方式，但必须配合沿触发、Busy/Done/Ack 或类似握手语义，才能真正稳定。

### 追问风险点

- 不要把“轮询”说成唯一方案
- 不要忽略轮询带来的延迟和重复消费问题
- 回答里最好顺手提一句“工业现场更看重交付和维护成本”

## 4. 软件触发、硬件触发、PLC 触发三者是什么关系

### 1 分钟版

软件触发和硬件触发描述的是相机怎么被拍，PLC 触发描述的是谁在发起这次节拍。PLC 可以通过 IO 直接硬触发相机，也可以通过通信告诉上位机去做软件触发。

### 3 分钟版

这三个概念最容易被混在一起。软件触发是上位机主动调用相机采图，适合调试和节拍要求不高的场景；硬件触发是外部电信号直接让相机拍照，更适合现场高节拍和时序要求强的工位。  
所谓 PLC 触发，本质上不是第三种并列的触发技术，而是说 PLC 在这次流程里扮演了发起者。如果 PLC 直接给相机触发线，那就是硬件触发；如果 PLC 只是通过通信给上位机一个 Ready 或 Trigger 状态，再由上位机调用采图，那还是软件触发。  
在 ClearVision 里，这件事可以回扣到 `ImageAcquisition.triggerMode` 和 `TriggerModule`。我会强调触发不是一个按钮，而是一段时序。

### 追问风险点

- 不要把“PLC 触发”说成和软件/硬件完全平级的新类别
- 不要忽视相机曝光与现场节拍的关系
- 回答里最好出现 `triggerMode`

## 5. Modbus 和厂商协议怎么选

### 1 分钟版

如果现场已经明确是三菱、西门子、欧姆龙这类厂牌，我会优先考虑对应原生协议；如果是多品牌设备统一接入，Modbus 往往更像公共语言；如果是自定义控制器或服务，再考虑 TCP。

### 3 分钟版

协议选择不是拼概念，而是看现场设备、维护团队和交付边界。如果 PLC 品牌已经确定，而且现场工程师也都按厂牌地址和工具在工作，那么原生协议通常更直接，比如三菱优先 MC、西门子优先 S7、欧姆龙优先 FINS。  
如果项目是多品牌设备整合，或者更强调开放和通用，那 Modbus TCP 往往更适合作为统一接口。再往下，如果要对接的是自定义控制器、工位服务或者结构化业务报文，就可以考虑 `TcpCommunication` 这种更灵活的方式。  
在 ClearVision 里，这一层能直接对应到 `MitsubishiMcCommunication`、`ModbusCommunication`、`SiemensS7Communication`、`OmronFinsCommunication` 和 `TcpCommunication` 这些算子。

### 追问风险点

- 不要把“开放协议”说成永远更优
- 不要把原生协议说成“更高级”，本质是“更贴现场”
- 记得回到 ClearVision 的对应算子

## 6. 怎么处理 PLC 触发拍照和 OK/NG 回写

### 1 分钟版

我会把这件事看成一个完整闭环：PLC 发起触发，上位机采图并完成视觉判定，`ResultJudgment` 先把业务结果收口，再通过通信算子回写 OK/NG 或测量值，最后最好有 Done/Ack 之类的确认，不把一次写值当成整个流程结束。

### 3 分钟版

最常见的闭环是：PLC 检测到工件到位，发出触发或允许检测状态；相机采图，上位机执行视觉流程；流程最后由 `ResultJudgment` 统一收口成 OK/NG、数值或状态码，再通过 `MitsubishiMcCommunication` 或 `ModbusCommunication` 回写给 PLC。  
这里最关键的不是“能写回去”，而是写回的时机和语义。比如结果值是 `1` 还是 `0`、什么时候写 Done、PLC 什么时候清 Trigger、上位机什么时候清 Busy，这些都属于时序设计。  
如果节拍高，我会更倾向让 PLC 直接硬件触发相机；如果节拍没那么紧，也可以采用 PLC 通过通信发状态、上位机再软件触发的方式。但不管哪种方式，业务结果最好先经过 `ResultJudgment`，而不是把原始算法输出直接暴露给 PLC。

### 追问风险点

- 不要忽略 Done / Ack / Busy 这类握手语义
- 不要把“写值成功”说成“闭环完成”
- 最好提到 `ResultJudgment`

## 7. 为什么“通信连通了”不代表“检测流程真的可靠”

### 1 分钟版

因为连通只说明某次值传过去了，不说明地址一定对、语义一定对、时机一定对，更不说明 PLC 已经在正确时间窗口里消费了结果。

### 3 分钟版

这是工业系统里很关键的边界。比如 TCP 连通了，但报文格式不对；寄存器写成功了，但写到了错误地址；值写对了，但 `1` 到底代表 OK 还是 NG 双方理解不一致；或者结果写回时 PLC 已经超时进入别的分支了。  
所以我在回答这类问题时会强调两层：第一层是通信链路成功，第二层是业务闭环可靠。这也是 ClearVision 历史文档里那条 `SAFETY_001` 很有价值的原因，它要求通信类算子上游最好先经过 `ConditionalBranch` 或 `ResultJudgment`，背后的思路就是“先明确业务含义，再做对外回写”。

### 追问风险点

- 不要把 `Status=true` 说成万事大吉
- 不要忽略地址、语义、时机三个维度
- 回答里提到 `SAFETY_001` 会很加分

## 8. 如果 PLC 连续触发两次、网络抖动或回写超时，系统该怎么兜底

### 1 分钟版

我的思路会围绕三件事：沿触发避免重复消费、超时和失败路径避免无限等待、断连后重连并重新同步关键状态。通信异常和业务 NG 要分开处理，不能混成一种结果。

### 3 分钟版

如果 PLC 连续触发两次，我会优先考虑上升沿消费、Busy 位和 Done/Ack 握手，避免同一个电平状态被反复消费。网络抖动时，不能只做无脑重试，还要考虑幂等性，否则可能把结果写两次。对于回写超时，系统不能无限阻塞，应该进入明确的异常路径，并把“系统异常”和“产品 NG”分开。  
如果连接真的断了，重连只是第一步，重连后还要重新读关键状态，确认 PLC 当前停在哪个节拍，不然很容易出现上位机以为还是上一拍、现场已经进入下一拍的错位。  
在 ClearVision 的表达里，这一层可以回扣到通信算子的 `PollingTimeout / PollingInterval` 这类参数语义，以及 `ResultJudgment` 做业务收口的必要性。

### 追问风险点

- 不要只说“加重试”
- 不要把系统异常都归成 NG
- 要体现“重连后重新同步状态”的意识

## 9. 上位机下发参数 / 配方给 PLC 时，为什么不能让 AI 自己猜

### 1 分钟版

因为现场配方、地址区、相机绑定、模型路径、阈值和 ROI 都是具体工位的真实资源，AI 可以给建议，但不能凭空知道现场最终配置，所以一定要保留人工确认和真实配置注入。

### 3 分钟版

这类问题特别适合把你的主线和工业系统观接起来。我的稳定说法是：AI 在 ClearVision 里可以帮助生成流程草案、推荐模板、提示待确认参数，但现场真正的配方和资源配置必须来自真实工位，而不是让模型猜。  
比如 PLC 参数区地址、相机 ID、ROI、模型文件路径、结果编码方式，这些都不是纯文本推理能可靠得出的。所以上位机可以做配方管理和下发，但最终要经过配置注入、Validator、DryRun 和人工确认。  
这样回答既说明你理解现场约束，也不会把 AI 包装成“全自动落地”。

### 追问风险点

- 不要说成“AI 当然可以全自动决定现场参数”
- 不要把“推荐”说成“确定”
- 要把话题拉回你的 LLM 编排主线

## 10. 在 ClearVision 里，这些算子各自承担什么职责

### 1 分钟版

我会这样归纳：`ImageAcquisition` 负责采图，`TriggerModule` 负责节拍组织，`ResultJudgment` 负责业务结果收口，`MitsubishiMcCommunication / ModbusCommunication / SiemensS7Communication / OmronFinsCommunication` 负责 PLC 通信，`TcpCommunication` 负责自定义控制系统通信。

### 3 分钟版

如果要把 ClearVision 讲成真实工业软件，我不会只背算子名字，而是讲职责分工。`ImageAcquisition` 是感知入口，和相机绑定、触发模式相关；`TriggerModule` 更像节拍组织器，用来承接软件触发、定时触发或外部信号思路；`ResultJudgment` 的作用不是做图像算法，而是把上游视觉结果收口成 PLC 更能消费的 OK/NG 或数值；多种通信算子则负责和不同设备或控制系统对接。  
我觉得这类回答的重点，是让面试官听出来：我知道这些算子在现场链路里的位置，也知道通信应该放在业务判定之后，而不是把它们当成孤立的胶水节点。

### 追问风险点

- 不要把 `ResultJudgment` 讲成图像算子
- 不要把 `TriggerModule` 讲成“PLC 通信模块”
- 回答要体现“职责分层”
