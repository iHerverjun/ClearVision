# 面试禁语与替代表述清单

> 用途：把容易失真的说法，替换成“能自证、能落地、能承受追问”的表达。

---

## 1. 使用原则

- 只要一句话说完后，你自己都知道拿不出证据，就不要说。
- 只要一句话默认暗示“已经成熟商用”，而实际还在原型阶段，就降级表述。
- 只要一句话把 AI 辅助说成了 AI 代替人工，就重写。

---

## 2. 禁语清单

| 禁语 | 为什么危险 | 替代表述 |
|---|---|---|
| `这是一个成熟商用平台。` | 当前更接近原型系统 / MVP。 | `这是一个已经跑通核心链路的工业视觉原型系统。` |
| `50 万行代码都是我独立手写的。` | 与实际 AI 辅助开发模式不符，且不可验证。 | `项目由我主导设计，并在 AI 辅助开发模式下快速实现和迭代。` |
| `160 个算子我都很熟。` | 很容易被深挖击穿。 | `我重点掌控的是主链路和少数主打算子，其余算子按 A/B/C 分级准备。` |
| `AI 自动生成后就能直接用。` | 忽略 Validator、DryRun、人工确认。 | `AI 先生成结构化流程草案，再经过校验、预演和人工确认。` |
| `工人完全零门槛就能用。` | 过度产品化承诺。 | `目标是降低搭流程门槛，但高风险参数和资源仍需人工确认。` |
| `我们比 VisionMaster / Halcon 快很多。` | 没有统一口径和真实证据时风险极高。 | `当前只和同场景手工搭建流程或仓库内 demo 版本做对比。` |
| `提效 100% 以上。` | 目前缺少完整统计证据。 | `目前有初步观察，正在以场景记录和证据台账补真实对比。` |
| `格式校验过了，说明流程是对的。` | 把 Schema 合法和业务正确混在一起。 | `格式合法只说明链路可解析，业务正确还需要样本和验收标准验证。` |
| `AI 知道相机 ID、模型路径、ROI。` | 这些配置通常来自现场或人工注入。 | `这些环境相关配置需要用户或模板侧显式提供。` |
| `如果 AI 不可用也没关系。` | 没有可执行 Plan B 时会显得风险意识不足。 | `我准备了模板优先、本地模型和人工确认优先的降级方案。` |
| `这个项目已经多场景稳定落地。` | 当前最强证据仍集中在线序等试点场景。 | `目前最完整的可自证场景是线序检测，其他场景还在继续补证据。` |
| `我对所有工程细节都掌控。` | 容易被 Bug、Profiling、单测追问击穿。 | `我最有掌控度的是 LLM 编排主线和部分关键工程链路。` |
| `那个问题其实没卡点。` | 会暴露风险意识不足。 | `那一块我当时做了取舍，优先保证链路可控，后续再继续收敛。` |
| `AI 不会错，错了再改。` | 工程态度危险。 | `AI 生成是提速手段，真正落地靠模板收缩、校验和人工审计。` |
| `我一个人全栈全都做完了。` | 太宽，可信度低。 | `我主导了关键链路设计，也承担了实现和验证中的主要工作。` |

---

## 3. 三个高频替换模板

### 3.1 把“夸大成果”改成“可自证阶段”

原说法：

> 我已经做出一个成熟工业视觉平台。

替换成：

> 我目前做出的是一个跑通了核心链路的工业视觉原型系统，重点验证的是 LLM 编排和模板化落地的可行性。

### 3.2 把“夸大个人代码量”改成“强调主导权”

原说法：

> 这些代码都是我一个人从零写完的。

替换成：

> 我主导了整体设计和关键链路验证，代码实现过程中大量使用了 AI 辅助开发，但关键边界和取舍由我负责。

### 3.3 把“AI 自动化神化”改成“AI + 约束 + 人工”

原说法：

> 用户说一句话，AI 就能自动把项目搭好。

替换成：

> 用户先描述需求，系统优先命中模板，再生成结构化流程草案；之后还要经过校验、预演和人工确认，才能进入运行时。
