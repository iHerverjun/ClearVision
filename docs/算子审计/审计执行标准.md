# 算子审计执行标准

## 1. 目的

- 本标准用于约束 `docs/算子审计/` 下所有计划、台账、证据链、优先级清单和周报的写法。
- 目标是把算子审计文档收敛成“单入口启动、自动回填、可追溯验收”的工程文档，避免继续膨胀成碎片化小计划集合。

## 2. 核心原则

- 用户只面向 1 份启动计划文件：`docs/算子审计/启动计划/WeekN-长时启动计划.md`。
- `docs/算子审计/` 根目录中的其余文档默认都是结果文档或索引文档，不再要求用户逐个打开判断执行顺序。
- 主计划的执行粒度只能是“长时工作块”，默认每块连续执行 `60-120 分钟`，复杂块可放宽到 `180 分钟`。
- 一份可执行的启动计划默认只保留 `4-5 个` 工作块，每个工作块都必须独立产出可验收工件。
- 只有当某一步的输出能被下游直接消费时，才允许拆成独立步骤。
- 启动计划必须直接回答 6 个问题：审什么、按什么工作块审、产出什么、自动写回哪里、凭什么算完成、当前做到哪一步。
- 未经证据核验的问题不得直接进入 Week2 修复清单。

## 3. 文档分工

| 文档 | 角色 | 允许内容 | 不允许内容 |
| --- | --- | --- | --- |
| `启动计划/WeekN-长时启动计划.md` | 用户启动入口 | 唯一执行入口、工作块、时长、依赖、暂停恢复、自动回填规则 | 周内结果明细回填 |
| `WeekN-并行审计计划.md` | 索引/兼容入口 | 指向启动计划、说明结果文档职责 | 重复承载长计划正文 |
| `WeekN-审计台账.md` | 主执行台账 | 冻结结论、静态盘点、实验设计、风险项 | 新的排程或候选计划 |
| `WeekN-已升级证据链.md` | 证据与复核台账 | 证据源、脚本入口、结果产物、状态复核 | 独立排期或新待办体系 |
| `WeekN-修复优先级清单.md` | 缺陷池与修复入口 | P0/P1 分级、阻塞项、Owner、截止日期 | 预置候选池、按天排程 |
| `WeekN-审计周报.md` | 验收与移交 | 本周结论、遗留、风险、Week2 入口状态 | 详细执行计划 |
| `TODO.md` | 总体进度板 | 分类状态、滚动周期、全局 DoD | 周内细颗粒动作排程 |

## 4. 自动回填模型

- 启动后由 Codex 按工作块自动维护结果文档，不再额外生成新的周中计划文档。
- `工作块A / 工作块B` 完成后，回填 `WeekN-审计台账.md` 的范围结论、实验入口、风险项和周内进度。
- `工作块C` 完成后，回填 `WeekN-已升级证据链.md` 的证据映射、复核状态、结果产物绑定和缺口说明。
- `工作块D` 完成后，回填 `WeekN-修复优先级清单.md` 的 P0/P1 缺陷池、阻塞清单和不入池结论。
- `工作块E` 完成后，回填 `WeekN-审计周报.md` 的周结论、风险遗留、Week2 入口状态；必要时同步 `TODO.md` 的全局分类状态。
- 除启动计划外，其他文档不再承担“告诉用户下一步做什么”的职责，只负责承接执行结果。

## 5. 标准工作块模型

| 工作块 | 典型时长 | 必做动作 | 直接产物 | 完成门槛 |
| --- | --- | --- | --- | --- |
| 工作块1：范围冻结与入口就绪 | 60-90 分钟 | 锁定类别、算子清单、责任线、引用规范、回填路径 | 启动计划 + 台账骨架 + 证据链骨架 | 范围无歧义，交付路径确定 |
| 工作块2：主审计包静态闭环 | 90-180 分钟 | 补实现、参数、依赖、异常路径、已知限制 | 台账中的静态盘点首版 | 目标范围静态覆盖率达到 100% |
| 工作块3：复现实验与证据链收口 | 90-180 分钟 | 为每个审计包给出最小复现集、脚本入口、结果映射 | 台账实验区 + 证据链映射表 | 每个审计包都有“脚本/测试 -> 样本 -> 结论”路径 |
| 工作块4：风险收敛与 Week2 入池 | 60-120 分钟 | 汇总 P0/P1、阻塞项、是否进入 Week2 | 修复优先级清单 | 每条问题均有证据、Owner、日期 |
| 工作块5：验收与移交 | 60-90 分钟 | 汇总结论、遗留、Week2 入口状态 | 审计周报 | 只有“已核验证据结论”可以对外发布 |

## 6. 标准计划写法

一份合格的启动计划至少包含以下 7 节：

| 节 | 必须回答的问题 |
| --- | --- |
| 使用方式 | 用户只需要做什么决定，如何启动和恢复 |
| 安全边界 | 哪些动作可以做，哪些情况必须暂停 |
| 工作块总览 | 分几个长时工作块、先后依赖和时长 |
| 工作块详情 | 每块覆盖什么、怎么连续推进、何时结束 |
| 自动回填 | 每块结束后写回哪些结果文档 |
| 当前状态 | 已完成什么，下一块是什么 |
| 验收门槛 | 什么情况下 Week2 可以入场 |

每个工作块至少要写清以下 8 个字段：

| 字段 | 要求 |
| --- | --- |
| 工作块名称 | 必须是可执行的闭环动作，不是日期标题 |
| 预估时长 | 默认 60-120 分钟，复杂块可放宽到 180 分钟 |
| 覆盖范围 | 必须能看出审计对象 |
| 启动前检查 | 必须说明开工前的安全和输入确认 |
| 连续执行内容 | 只写同一上下文内连续推进的动作 |
| 输出回填 | 必须指定写回哪份结果文档 |
| 完成闸门 | 必须能判断这一块是否真的做完 |
| 暂停条件 | 必须说明什么情况下应暂停并等待确认 |

## 7. 标准记录写法

每条审计结论必须包含以下字段：

| 字段 | 说明 |
| --- | --- |
| 结论 | 只写已核验证据结论 |
| 证据路径 | 必须到仓库路径，优先到行号级 |
| 复现方式 | 脚本、测试类、样本或最小手工步骤 |
| 风险/阻塞 | `P0 / P1 / P2` 或 `数据 / 模型 / 硬件 / 依赖` |
| 负责人 | 必须可追责 |
| 截止日期 | 必须可执行 |

## 8. 禁止项

- 禁止再为同一周新增“子计划”“候选计划”“补充排期”类文档。
- 禁止把 10-30 分钟的零碎动作包装成独立计划块。
- 禁止在 `WeekN-修复优先级清单.md` 中预置“候选项”“待入池项”“可能问题列表”。
- 禁止把台账写成新的周计划，把周报写成新的待办列表。
- 禁止把“待确认”问题直接带入 Week2 修复清单。
- 禁止只写口头判断，不写证据路径和复现入口。

## 9. 本周落地要求

- 从本周开始，用户统一从 `docs/算子审计/启动计划/Week1-长时启动计划.md` 启动 Week1 审计。
- `Week1-并行审计计划.md` 仅保留为索引入口，不再承载详细计划正文。
- 已存在的台账、证据链、优先级清单、周报继续保留，但统一视为自动回填的结果文档。
- 后续新增内容默认写入现有结果文档，不再新建更多计划性文档。
