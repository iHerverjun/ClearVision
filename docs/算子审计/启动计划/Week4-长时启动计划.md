# Week4 长时启动计划

> 这是 Week4 算子审计/验收收口的唯一启动入口。Week4 仅承接 Week3 已确认且未解除的唯一外部阻塞：`AI检测验收补证（工业签收闭环 + 模型唯一标识/环境实值补齐）`，不重开 Week2/Week3 已审结范围。
>
> 本计划主执行窗口固定为 2026-04-27 至 2026-05-01（UTC+8）；已于 2026-04-13 19:32（UTC+8）按用户指令提前启动，并完成 C1-C4 当次收口执行。

## 1. 使用方式

- 全量启动：直接告诉 Codex “现在按 Week4 长时启动计划执行”。
- 定时启动：直接告诉 Codex “2026-04-27 09:00 按 Week4 长时启动计划执行”。
- 断点恢复：直接告诉 Codex “从工作块C2继续执行” 或 “从一致性核验开始执行”。
- 默认执行链：`工作块C1 -> 工作块C2 -> 工作块C3 -> 工作块C4`。
- 除“开始时间”和“是否从某块恢复”外，用户不需要再手工拆任务或指定回填路径。

## 2. 安全边界

- Week4 目标是“关闭 Week3 已确认单条外部验收阻塞”，不是“新增修复开发范围”。
- Week4 只允许推进 1 条阻塞：`AI检测验收补证`。
- 任何状态转 `[x]` 必须同时满足：`批次结果 + 模型版本 + 阈值 + 环境 + 工业签收`，且与同一批次结果一一绑定。
- 不得重开 Week2 A/B 已审结项（预处理 6 项、`EdgeDetection` 直单测缺口）。
- 同一 `.csproj` 不允许并发运行多个 `dotnet test`；如需定向补跑，统一在当前 PowerShell 会话使用 `& "./scripts/run-dotnet-test-serial.ps1"` 串行执行；如需整包补跑，仅允许一次 `scripts/run-tests-detection-regression.ps1 -Gate all`。
- 不执行破坏性 Git 操作，不覆盖用户未明确授权的现有改动。
- 如果证据材料缺失或口径冲突，允许暂停当前工作块，但必须先把“缺什么、卡在哪里、谁负责、下次检查时间”写回结果文档。

## 5. 工作块详情

### 5.1 工作块C1：启动冻结与责任链续签

- 目标：在 Week4 开始时把执行对象、责任链和截止口径一次性续签，避免周内反复改口径。
- 启动前检查：
  1. `Week3-修复优先级清单.md` 中仅保留 1 条 `AI检测验收补证` 阻塞。
  2. 阻塞项包含 `Owner + 截止日期 + 解除动作 + 升级路径`。
  3. `Week2-已升级证据链.md` 中已存在冻结批次与 Week3 核验结论。
- 连续执行内容：
  1. 锁定 Week4 范围：仅推进单条验收阻塞，不新增 A/B 或新阻塞。
  2. 续签外部 Owner、审核人、升级触发时间。
  3. 校准 Week4 检查点（周内中段检查 + 周五收口检查）。
- 输出回填：写回 `Week3-修复优先级清单.md` 的阻塞项说明与检查点时间。
- 完成闸门：阻塞定义、责任链、时间口径三者一致，且无冲突字段。
- 暂停条件：Owner 不明确、截止日期冲突、阻塞项存在多重解释。

### 5.2 工作块C2：外部材料追齐与映射补全

- 目标：补齐 AI 检测 6 算子的验收证据，并完成“批次 -> 模型 -> 阈值 -> 环境 -> 签收”一一绑定。
- 启动前检查：
  1. 已有冻结批次产物：`test_results/detection-all-20260413-115808.trx`。
  2. 已有补跑入口：`scripts/run-tests-detection-regression.ps1 -Gate all`。
  3. 证据模板字段统一：模型唯一标识、阈值实值、环境实值、签收记录。
- 连续执行内容：
  1. 补齐 6 算子的模型标识与版本，明确缺省项与来源位。
  2. 补齐阈值配置值与运行时实值，绑定到同一批次。
  3. 补齐环境实值（代码版本、依赖、推理后端、GPU/驱动、时间窗）。
  4. 补齐工业签收实录（签收人、签收时间、签收范围、签收结论、附件索引）。
- 输出回填：写回 `Week2-已升级证据链.md` 第 5 节模板与周内记录区。
- 完成闸门：5 类材料齐全且可追溯，映射关系可复核。
- 暂停条件：材料迟交、版本冲突、签收记录不可追溯。

### 5.3 工作块C3：一致性核验与条件补跑

- 目标：确认证据与批次结果一致；必要时补跑一次并更新映射。
- 启动前检查：
  1. C2 材料已完整回填。
  2. 具备一致性核验口径：同批次、同模型、同阈值、同环境。
  3. 若触发补跑，保证同一 `.csproj` 测试串行执行。
- 连续执行内容：
  1. 执行一致性核验：脚本入口、批次结果、模型/阈值/环境、签收结论四段对齐。
  2. 做补跑判定：一致则不补跑；不一致则串行触发一次 `scripts/run-tests-detection-regression.ps1 -Gate all`。
  3. 如生成新批次，更新映射并触发签收重确认。
- 输出回填：写回 `Week2-已升级证据链.md`（核验结论）与 `Week3-修复优先级清单.md`（阻塞状态）。
- 完成闸门：形成单一审核结论（可转 `[x]` 或继续 `[-]`），并给出明确依据。
- 暂停条件：结果与证据冲突、环境漂移导致不可复现、签收口径无法对齐。

### 5.4 工作块C4：收口发布与 Week5 入口

- 目标：输出 Week4 最终结论，并把未解除阻塞收敛为可追责的单条移交卡片。
- 启动前检查：
  1. C3 已给出可审签结论。
  2. `Week3-修复优先级清单.md` 与 `Week2-已升级证据链.md` 口径一致。
  3. 周级摘要文档可承接最终状态。
- 连续执行内容：
  1. 如条件满足，将阻塞状态转 `[x]` 并写明完整证据链。
  2. 如条件未满足，保留 `[-]` 并写明唯一缺口、Owner、截止日期、解除动作、升级路径。
  3. 同步 Week4 周结论到周报，并给出 Week5 入口条件。
- 输出回填：写回 `Week3-修复优先级清单.md` 与 `Week2-审计周报.md`。
- 完成闸门：文档状态一致且结论可独立阅读、可追责、可接续。
- 暂停条件：文档结论冲突、签收未闭环但被要求提前关单。

- G1：证据完整（5 类材料齐全）。
- G2：证据一致（与批次结果无冲突）。
- G3：签收闭环（签收信息可追溯）。
- G4：文档一致（清单、证据链、周报状态一致）。
