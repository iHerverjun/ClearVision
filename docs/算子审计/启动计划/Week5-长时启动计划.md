# Week5 长时启动计划

> 这是 P1/P2 分类总体修复的唯一启动入口。它承接 `TODO.md` 中尚未闭环的 P1、P2 分类，并与 `Week5-入口卡片.md` 中的 AI 检测外部阻塞并行跟踪，但不把该外部阻塞扩散为新的内部修复范围。
>
> 本计划的建议执行窗口为 2026-05-04 至 2026-05-15（UTC+8）。其中 `2026-05-04 ~ 2026-05-08` 主做 P1 复核与主审计，`2026-05-11 ~ 2026-05-15` 主做 P2 快审、遗留收敛与全库发布建议；已于 2026-04-14 09:56（UTC+8）按用户指令提前启动，并完成 D1 范围冻结与结果骨架就绪；已于 2026-04-14 12:01（UTC+8）完成 D2 历史收口复核。

## 1. 使用方式

- 全量启动：直接告诉 Codex “现在按 Week5 长时启动计划执行”。
- 定时启动：直接告诉 Codex “2026-05-04 09:00 按 Week5 长时启动计划执行”。
- 断点恢复：直接告诉 Codex “从工作块D3继续执行” 或 “从 P2 快审开始执行”。
- 默认执行链：`工作块D1 -> 工作块D2 -> 工作块D3 -> 工作块D4 -> 工作块D5`。
- 除“开始时间”和“是否从某块恢复”外，用户不需要再手工拆任务或指定回填路径。

## 2. 安全边界

- Week5/Week6 目标是“完成 P1/P2 分类的工业级复核、补测、修复收敛和发布建议”，不是把所有历史遗留一次性扩成开发大会战。
- `Week5-入口卡片.md` 中的 AI 检测单条外部阻塞继续独立跟踪；它不阻塞 P1/P2 的内部审计执行，但会阻塞“全库已完成工业级签收”的最终对外表述。
- Region / Morphology / Frequency 这 3 个历史已收口 P1 分类，必须先按当前口径复核，不能直接视为 `[x]`。
- 数据处理 / 流程控制 / 逻辑工具 / 输出 / 通信 / 采集 / 3D 必须先形成“实现审计 -> 复现入口 -> 风险分级 -> 首批修复池”闭环，才允许进入最终发布建议。
- P2 分类默认按“低耦合专项能力补齐 + 工程可维护性补齐”处理，除非证据明确指向 P0/P1 风险，否则不升级范围。
- 同一 `.csproj` 不允许并发运行多个 `dotnet test`；如需定向回归，统一在当前 PowerShell 会话使用 `& "./scripts/run-dotnet-test-serial.ps1"` 串行执行。
- 不执行破坏性 Git 操作，不覆盖用户未明确授权的现有改动。
- 如果发现某分类缺少样本、脚本入口或历史产物，允许暂停当前工作块，但必须先把“缺什么、卡在哪里、谁负责、下次检查点”写回结果文档。

## 6. 工作块详情

### 6.1 工作块D1：范围冻结与结果骨架就绪

- 目标：把 P1/P2 总体修复范围、责任线和回填路径一次性冻结，避免后续在执行中反复换口径。
- 启动前检查：
  1. `TODO.md` 中 P1/P2 分类和优先级定义可直接引用。
  2. `Week5-入口卡片.md` 中 AI 检测外部阻塞已明确为“并行跟踪，不内化扩范围”。
  3. 当前仓库没有其他与 P1/P2 同名的新计划文件正在承载并行排程。
- 连续执行内容：
  1. 把 P1/P2 拆成“历史收口复核 / 主功能审计 / 现场链路审计 / 专项能力快审 / 工程维护快审”五组。
  2. 创建或刷新 Week5 结果台账、证据链、Week6 修复清单和 Week5 周报骨架。
  3. 绑定 Owner、检查点和默认回填路径。
- 输出回填：写回 `Week5-审计台账.md` 的范围冻结区与结果骨架说明。
- 完成闸门：范围、责任线、回填路径三者一致，且没有与现有 Week2-Week5 文档冲突。
- 暂停条件：分类范围不一致、结果文档命名冲突、外部阻塞被错误纳入内部修复范围。

### 6.2 工作块D2：P1 历史收口复核

- 目标：把 Region / Morphology / Frequency 从“历史已收口、当前待复核”推进到“可转 `[x]` 或明确维持 `[-]`”。
- 启动前检查：
  1. 可定位历史证据、脚本入口或结果产物。
  2. 当前口径下的 DoD 已明确：精度/稳定性/性能/回归/遗留清单。
  3. 如需补跑，确认同一 `.csproj` 测试不会并发。
- 连续执行内容：
  1. 逐类核对历史证据是否覆盖当前口径。
  2. 为每类补“脚本/测试 -> 结果产物 -> 文档结论”映射。
  3. 明确缺口属于脚本、样本、环境、产物还是结论口径缺失。
- 输出回填：写回 `Week5-已升级证据链.md` 和 `Week5-审计台账.md`。
- 完成闸门：3 个分类都给出当前状态结论，且具备可追溯证据路径。
- 暂停条件：历史产物缺失、脚本不可复现、现口径与旧口径无法对齐。

### 6.3 工作块D3：P1 主功能与现场链路主审计

- 目标：完成 7 个未启动 P1 分类的首轮工业级审计，把问题收敛为正式修复池，而不是停留在“待看一眼”。
- 启动前检查：
  1. 分类实现路径、算子清单和依赖链可定位。
  2. 需要测试的入口可定位，且满足串行规则。
  3. 已定义主审计输出字段：实现审计、复现入口、异常路径、性能边界、风险分级。
- 连续执行内容：
  1. 对数据处理 / 流程控制 / 逻辑工具 / 输出做主功能静态审计，重点抓正确性、失败收口、对外契约。
  2. 对通信 / 采集 / 3D 做现场链路审计，重点抓联动稳定性、时序一致性、硬件/性能边界。
  3. 为每类补最小复现入口，并标记自动化覆盖缺口。
  4. 输出首批正式问题池候选和不入池结论。
- 输出回填：写回 `Week5-审计台账.md`。
- 完成闸门：7 个分类均形成首版审计结论，且每类至少有 1 条可执行复现入口或明确缺口说明。
- 暂停条件：依赖链缺失、无法复现、外部系统不可用、硬件口径不明。

### 6.4 工作块D4：P2 快审与工程维护补齐

- 目标：在不无谓放大范围的前提下，完成 P2 分类的最小工业级快审，为 Week6 发布建议提供“能否延后、能否豁免、哪些必须入池”的依据。
- 启动前检查：
  1. D3 已形成统一问题记录口径。
  2. P2 分类的代码入口和典型调用场景可定位。
  3. 已定义 P2 输出重点：适用边界、维护成本、组合稳定性、专项能力缺口。
- 连续执行内容：
  1. 对 Texture / Analysis / AI Inspection 做专项能力快审，记录适用边界和不可替代性。
  2. 对变量 / 通用 / 辅助 / 拆分组合做工程维护快审，记录接口复用、异常收口和组合退化风险。
  3. 为确需入池的问题补最小复现和责任位；其余形成不入池结论。
- 输出回填：写回 `Week5-审计台账.md` 和 `Week5-已升级证据链.md`。
- 完成闸门：7 个 P2 分类均形成“入池 / 不入池 / 延后”三选一结论，且有依据。
- 暂停条件：调用场景缺失、样本不可得、问题无法复现或无法界定影响范围。

### 6.5 工作块D5：风险收敛、正式入池与发布建议

- 目标：把 P1/P2 的已核验问题收敛成 Week6 可执行修复池，并给出是否能对外描述为“接近全库工业级”的发布建议。
- 启动前检查：
  1. D2/D3/D4 都已形成正式结论。
  2. 所有入池问题具备 `确认结论 + 证据路径 + 复现方式 + 风险等级 + 负责人 + 截止日期`。
  3. `Week5-入口卡片.md` 中的 AI 检测阻塞状态已同步到最终建议口径。
- 连续执行内容：
  1. 汇总 P1/P2 已核验问题，按 P1/P2、内部修复/外部阻塞、可立即修复/需专项排期分层。
  2. 写入 `Week6-修复优先级清单.md` 的正式缺陷池、阻塞清单和不入池结论。
  3. 写入 `Week5-审计周报.md` 的阶段结论与发布建议。
  4. 同步 `TODO.md` 的分类状态和后续阶段入口。
- 输出回填：写回 `Week6-修复优先级清单.md`、`Week5-审计周报.md`、`TODO.md`。
- 完成闸门：修复清单、周报、TODO 三方状态一致，且“全库是否可对外宣称工业级”给出单一口径。
- 暂停条件：文档结论冲突、入池问题仍缺证据、AI 检测阻塞被忽略或误判为已解除。
