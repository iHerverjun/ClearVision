# Week1 长时启动计划

> Week1 已闭环完成；遗留仅按 Week2 清单和 Week2 启动计划继续推进，本文件保留为历史入口记录。

> 这是 Week1 算子审计的唯一启动入口。你只需要决定什么时候执行；一旦启动，Codex 按本文件的工作块顺序推进，并自动维护台账、证据链、修复清单和周报。

## 1. 使用方式

- 全量启动：直接告诉 Codex “现在按 Week1 长时启动计划执行”。
- 定时启动：直接告诉 Codex “今天 20:00 按 Week1 长时启动计划执行”。
- 断点恢复：直接告诉 Codex “从工作块C继续执行” 或 “从风险收敛开始执行”。
- 默认执行链：`工作块A / 工作块B / 工作块C` 并行推进，随后执行 `工作块D`，最后执行 `工作块E`。
- 除“开始时间”和“是否从某块恢复”外，用户不需要再手工拆任务或指定回填路径。

## 2. 安全边界

- 审计目标是“形成证据闭环”，不是“追求修复数量”。
- 未经证据核验的问题不得进入 `Week2-修复优先级清单.md`。
- 同一 `.csproj` 不允许并发运行多个 `dotnet test`；如需聚合测试，统一使用 `./scripts/run-dotnet-test-serial.ps1`。
- 不执行破坏性 Git 操作，不覆盖用户未明确授权的现有改动。
- 如果发现工作区脏变更与当前审计结论直接冲突，必须暂停并先确认。
- 如果缺少样本、脚本、环境信息或结果产物，允许暂停当前工作块，但必须先把“缺什么、卡在哪里、回填到哪”记录到结果文档。

## 5. 工作块详情

### 5.1 工作块A：预处理 P0 闭环审计

- 目标：把 `预处理(23)` 从“静态首版盘点”推进到“具备最小复现入口和初版风险分级”的可执行状态。
- 启动前检查：
  1. `Week1-审计台账.md` 中 A 线范围冻结和静态首版已存在。
  2. 预处理相关测试入口、名片文档、实现路径可定位。
  3. 若需要测试，确认不与同一 `.csproj` 的其他测试并发。
- 连续执行内容：
  1. 按 A1/A2/A3 分组补齐 `正常 / 边界 / 异常` 最小实验集。
  2. 核对输入一致性、阈值敏感、几何对齐和性能预算风险。
  3. 标记直接单测缺口、复现缺口和只能手工复现的条目。
  4. 形成“算子 -> 复现入口 -> 问题结论 -> 风险等级”的闭环记录。
- 输出回填：写回 `Week1-审计台账.md` 的 A 线结果区和周内进度区。
- 完成闸门：23 个预处理算子全部具备复现入口或明确缺口说明，且形成可用于 D 块排序的初版风险结论。
- 暂停条件：缺少关键样本、测试入口冲突、实现与文档口径明显不一致。

### 5.2 工作块B：定位与特征主链稳定性闭环

- 目标：把 `定位(7) + 特征提取(4)` 从静态首版推进到“主链稳定性和联动回归入口清晰”的状态。
- 启动前检查：
  1. `Week1-审计台账.md` 中 B 线范围冻结和静态首版已存在。
  2. 相关上游依赖链已可定位，尤其是阈值链、边缘链、轮廓链。
  3. 需要运行测试时，遵守串行执行规则。
- 连续执行内容：
  1. 梳理 B1/B2/B3 三组主链、边缘链、修正链的输入依赖和断点。
  2. 为关键链路补“最小联动样本 -> 复现入口 -> 回归建议”。
  3. 记录错误收口方式、退化场景和无法自动化覆盖的边界。
  4. 形成“链路问题 -> 触发条件 -> 复现方式 -> 建议回归集”的闭环。
- 输出回填：写回 `Week1-审计台账.md` 的 B 线结果区和周内进度区。
- 完成闸门：11 个算子均被纳入主链稳定性分析，关键链路具备联动回归入口或明确缺口说明。
- 暂停条件：上游依赖缺失、链路样本无法复现、测试环境与静态结论不一致。

### 5.3 工作块C：补证与复核收口

- 目标：把已升级类和复核类从“有历史证据”推进到“本周可核验、可追溯”的状态。
- 启动前检查：
  1. `Week1-已升级证据链.md` 中证据源索引、复核包、映射表已存在。
  2. 检测、识别、标定、匹配定位、AI 检测的脚本或测试入口可定位。
  3. 若需要回归，确认产物落点和测试批次命名规则。
- 连续执行内容：
  1. 对检测/识别/标定补齐“脚本入口 -> 结果产物 -> 文档结论”映射。
  2. 对匹配定位和 AI 检测做本周状态复核，不提前把历史“已升级”直接视为 `[x]`。
  3. 记录模型、阈值、环境、产物批次等会影响复核结论的上下文。
  4. 明确哪些缺口是脚本缺口、样本缺口、产物缺口或验收签收缺口。
- 输出回填：写回 `Week1-已升级证据链.md` 的证据源、复核包、映射表和当前缺口。
- 完成闸门：每个补证/复核包都具备真实证据路径、复现入口、结果产物落点和当前状态结论。
- 暂停条件：缺少回归产物、模型/阈值/环境信息不完整、现有脚本无法稳定运行。

### 5.4 工作块D：风险收敛与 Week2 入池

- 目标：把 A/B/C 三块的已核验问题收敛成可直接交给 Week2 执行的正式问题池。
- 启动前检查：
  1. A/B/C 至少完成核心结论和缺口归档。
  2. 需要入池的问题都具备证据路径和复现方式。
  3. Owner、截止日期和阻塞类型具备可追责信息。
- 连续执行内容：
  1. 汇总 A/B/C 的已核验问题，去重、分级、排序。
  2. 区分正式入池问题、阻塞项和不入池但已审结事项。
  3. 绑定 `Owner + 截止日期 + 解除动作`。
  4. 清理掉所有“待确认”“待补证”项，确保它们不提前进入 Week2。
- 输出回填：写回 `Week2-修复优先级清单.md` 的缺陷池、阻塞清单和不入池结论。
- 完成闸门：每条入池问题都具备确认结论、证据、复现方式、负责人和日期。
- 暂停条件：证据不足、Owner 未绑定、问题仍停留在推测层。

### 5.5 工作块E：周验收与移交

- 目标：把 Week1 已完成闭环、遗留风险和 Week2 入口状态统一成一份可以独立阅读的验收摘要。
- 启动前检查：
  1. D 块已经完成正式入池。
  2. 台账、证据链、优先级清单中的状态相互一致。
  3. 周报中不存在未核验结论。
- 连续执行内容：
  1. 汇总本周完成的静态审计、复现实验、证据复核和风险收敛结果。
  2. 明确未完成项、暂停原因和 Week2 进入条件。
  3. 同步全局进度板需要更新的分类状态和阶段状态。
- 输出回填：写回 `Week1-审计周报.md`，必要时同步 `TODO.md`。
- 完成闸门：周报可独立阅读，且与台账、证据链、优先级清单结论一致。
- 暂停条件：上下游文档状态冲突、仍混入未核验推测结论。

## 6. 自动回填与进度维护规则

| 触发时机 | Codex 自动动作 | 回填目标 |
| --- | --- | --- |
| 工作块启动时 | 将该块标记为 `[-]`，写入当前执行重心和最近更新时间 | 启动计划 + 对应结果文档的进度区 |
| 工作块中段检查点 | 在不新建计划文档的前提下，把已确认结果直接补入目标文档 | 台账或证据链 |
| 工作块完成时 | 将该块标记为 `[x]`，写入完成结论、缺口、下一块入口 | 启动计划 + 对应结果文档 |
| D 块完成时 | 生成正式 P0/P1 问题池和阻塞项，不保留候选池 | 修复优先级清单 |
| E 块完成时 | 生成周验收摘要，并只保留已核验证据结论 | 审计周报 + 必要时同步 TODO |

- 默认不再新建周中补充计划、候选清单或按天排程。
- 如果执行被打断，下一次恢复直接从本文件状态板中最后一个 `[-]` 工作块继续。
