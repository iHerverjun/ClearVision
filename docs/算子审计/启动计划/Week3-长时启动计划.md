# Week3 长时启动计划

> 这是 Week3 算子审计/验收收口的唯一启动入口。Week3 仅处理 Week2 已确认的唯一外部阻塞：`AI检测验收补证`，不重开 Week2 已审结范围。
>
> 本计划主执行窗口固定为 2026-04-20 至 2026-04-24（UTC+8）；已于 2026-04-13 17:14（UTC+8）按用户指令提前启动 C1/C2 准备动作。

## 1. 使用方式

- 全量启动：直接告诉 Codex “现在按 Week3 长时启动计划执行”。
- 定时启动：直接告诉 Codex “2026-04-20 09:00 按 Week3 长时启动计划执行”。
- 断点恢复：直接告诉 Codex “从工作块C2继续执行” 或 “从一致性核验开始执行”。
- 默认执行链：`工作块C1 -> 工作块C2 -> 工作块C3 -> 工作块C4`。
- 除“开始时间”和“是否从某块恢复”外，用户不需要再手工拆任务或指定回填路径。

## 2. 安全边界

- Week3 目标是“关闭已确认外部验收阻塞”，不是“新增修复开发范围”。
- Week3 只允许推进 1 条阻塞：`AI检测验收补证`。
- 任何状态转 `[x]` 必须同时满足：`批次结果 + 模型版本 + 阈值 + 环境 + 工业签收`，且与同一批次结果一一绑定。
- 不得重开 Week2 A/B 已审结项（预处理 6 项、`EdgeDetection` 直单测缺口）。
- 同一 `.csproj` 不允许并发运行多个 `dotnet test`；如需补跑，统一使用 `scripts/run-tests-detection-regression.ps1 -Gate all`，并保持串行执行。
- 不执行破坏性 Git 操作，不覆盖用户未明确授权的现有改动。
- 如果证据材料缺失或口径冲突，允许暂停当前工作块，但必须先把“缺什么、卡在哪里、谁负责、下次检查时间”写回结果文档。

## 5. 工作块详情

### 5.1 工作块C1：启动冻结与责任链确认

- 目标：在 Week3 开始时把执行对象、责任链和截止口径一次性冻结，避免周内反复改口径。
- 启动前检查：
  1. `Week3-修复优先级清单.md` 中仅保留 1 条 AI 检测验收阻塞。
  2. 阻塞项包含 `Owner + 截止日期 + 解除动作 + 升级路径`。
  3. `Week2-已升级证据链.md` 中已存在初始批次 `detection-all-20260413-115808.trx`。
- 连续执行内容：
  1. 锁定 Week3 范围：仅推进 AI 检测验收补证，不重开 A/B。
  2. 确认外部 Owner、审核人、升级触发时间。
  3. 校准日期基线（升级检查点与最终结论日）。
- 输出回填：写回 `Week3-修复优先级清单.md` 的阻塞项说明与时间口径。
- 完成闸门：阻塞定义、责任链、时间口径三者一致，且无冲突字段。
- 暂停条件：Owner 不明确、截止日期冲突、阻塞项存在多重解释。

### 5.2 工作块C2：证据补齐与映射绑定

- 目标：补齐 AI 检测 6 算子的验收证据，并完成“批次 -> 模型 -> 阈值 -> 环境 -> 签收”一一绑定。
- 启动前检查：
  1. 已有初始批次产物：`test_results/detection-all-20260413-115808.trx`。
  2. 可定位 AI 检测回归入口：`scripts/run-tests-detection-regression.ps1 -Gate all`。
  3. 证据模板字段已统一：模型标识、阈值参数、环境上下文、签收记录。
- 连续执行内容：
  1. 收集模型版本材料（含唯一标识）并绑定到 6 算子。
  2. 收集阈值材料（配置值 + 运行时值）并绑定批次。
  3. 收集环境材料（代码版本、依赖、推理后端/GPU/驱动、时间窗口）。
  4. 收集工业签收记录（签收人、签收时间、范围、结论、附件索引）。
- 输出回填：写回 `Week2-已升级证据链.md` 第 2/3/4 节与周内记录区。
- 完成闸门：5 类材料齐全且可追溯，映射关系可复核。
- 暂停条件：材料迟交、版本冲突、签收记录不可追溯。

### 5.3 工作块C3：一致性核验与必要回归

- 目标：确认证据与批次结果一致，必要时补跑回归并更新绑定。
- 启动前检查：
  1. C2 的材料已完整回填。
  2. 已具备一致性核验口径：同批次、同模型、同阈值、同环境。
  3. 若触发补跑，保证同一 `.csproj` 测试串行执行。
- 连续执行内容：
  1. 执行一致性核验：脚本入口、批次结果、模型/阈值/环境、签收结论四段对齐。
  2. 做补跑判定：一致则不补跑；不一致则执行一次 `scripts/run-tests-detection-regression.ps1 -Gate all`。
  3. 若生成新批次，更新映射并触发签收重确认。
- 输出回填：写回 `Week2-已升级证据链.md`（核验结论）与 `Week3-修复优先级清单.md`（阻塞状态）。
- 完成闸门：形成单一审核结论（可转 `[x]` 或继续 `[-]`），且给出确切依据。
- 暂停条件：结果与证据冲突、环境漂移导致不可复现、签收口径无法重对齐。

### 5.4 工作块C4：收口发布与 Week4 入口

- 目标：输出 Week3 最终结论，并把未解除阻塞收敛为可追责的单条移交卡片。
- 启动前检查：
  1. C3 已给出可审签结论。
  2. `Week3-修复优先级清单.md` 与 `Week2-已升级证据链.md` 口径一致。
  3. 周级摘要文档可承接最终状态。
- 连续执行内容：
  1. 如条件满足，将阻塞状态转 `[x]` 并写明完整证据链。
  2. 如条件未满足，保留 `[-]` 并写明唯一缺口、Owner、截止日期、解除动作、升级路径。
  3. 同步 Week3 周结论到周报，并给出 Week4 入口条件。
- 输出回填：写回 `Week3-修复优先级清单.md` 与 `Week2-审计周报.md`。
- 完成闸门：文档状态一致且结论可独立阅读、可追责、可接续。
- 暂停条件：文档结论冲突、签收未闭环但被要求提前关单。

## 6. 自动回填与进度维护规则

| 触发时机 | Codex 自动动作 | 回填目标 |
| --- | --- | --- |
| 工作块启动时 | 将该块标记为 `[-]`，写入执行重心、Owner、最近更新时间 | 启动计划 + 对应结果文档 |
| 工作块中段检查点 | 回填本次新增证据、当前未过闸门（G1-G4）、下一Owner | 证据链或优先级清单 |
| 工作块完成时 | 将该块标记为 `[x]`，写入结论、缺口、下一块入口 | 启动计划 + 对应结果文档 |
| 阻塞未解除时 | 保留 `[-]`，必须写清升级触发时间和截止日期 | `Week3-修复优先级清单.md` |
| C4 完成时 | 形成 Week3 结论快照和 Week4 入口状态 | `Week2-审计周报.md` |

G1-G4 闸门定义：
- G1：证据完整（5 类材料齐全）。
- G2：证据一致（与批次结果无冲突）。
- G3：签收闭环（签收信息可追溯）。
- G4：文档一致（清单、证据链、周报状态一致）。
